# Згенеровані значення

Значення стовпців бази даних можуть генеруватися різними способами: стовпці первинного ключа часто є автоматично збільшуваними цілими числами, інші стовпці мають значення за замовчуванням або обчислені значення тощо. На цій сторінці детально описано різні шаблони генерації значень конфігурації за допомогою EF Core.

## Значення за замовчуванням

У реляційних базах даних стовпець можна налаштувати зі значенням за замовчуванням; якщо рядок вставлено без значення для цього стовпця, буде використано значення за замовчуванням.

Ви можете налаштувати значення за замовчуванням для властивості:

```cs
    modelBuilder.Entity<Blog>()
        .Property(b => b.Rating)
        .HasDefaultValue(3);
```
Ви також можете вказати фрагмент SQL, який використовується для обчислення значення за замовчуванням:

```cs
    modelBuilder.Entity<Blog>()
        .Property(b => b.Created)
        .HasDefaultValueSql("getdate()");
```
Значення за замовчуванням також набуває чинності під час додавання нового стовпця до таблиці; для існуючих рядків, які ще не мають значення для нового стовпця, буде встановлено значення за замовчуванням.

## Назва обмеження значення за замовчуванням

Починаючи з EF 10, для SQL Server ви можете явно вказати назву для обмежень значення за замовчуванням, що надає вам більше контролю над схемою бази даних.

```cs
modelBuilder.Entity<Blog>()
    .Property(b => b.Rating)
    .HasDefaultValue(3, "DF_Blog_IsActive");
```
```cs
modelBuilder.Entity<Blog>()
    .Property(b => b.Created)
    .HasDefaultValueSql("getdate()" , "DF_Blog_IsActive");
```
Ви також можете викликати UseNamedDefaultConstraints, щоб увімкнути автоматичне найменування всіх обмежень за замовчуванням. Зверніть увагу, що якщо у вас є існуючі міграції, то наступна додана міграція перейменує кожне обмеження за замовчуванням у вашій моделі.

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.UseNamedDefaultConstraints();
}
```

## Обчислювані стовпці

У більшості реляційних баз даних стовпець можна налаштувати так, щоб його значення обчислювалося в базі даних, зазвичай за допомогою виразу, що посилається на інші стовпці:

```cs
modelBuilder.Entity<Person>()
    .Property(p => p.DisplayName)
    .HasComputedColumnSql("[LastName] + ', ' + [FirstName]");
```
Вищезазначене створює віртуальний обчислюваний стовпець, значення якого обчислюється щоразу, коли воно отримується з бази даних. Ви також можете вказати, що обчислюваний стовпець має зберігатися (іноді його називають постійним), тобто він обчислюється під час кожного оновлення рядка та зберігається на диску разом зі звичайними стовпцями:

```cs
modelBuilder.Entity<Person>()
    .Property(p => p.NameLength)
    .HasComputedColumnSql("LEN([LastName]) + LEN([FirstName])", stored: true);
```

## Первинні ключі

За домовленістю, нескладені первинні ключі типу short, int, long або Guid налаштовуються таким чином, щоб для вставлених сутностей генерувалися значення, якщо значення не надається програмою. Ваш постачальник бази даних зазвичай бере на себе необхідну конфігурацію; наприклад, числовий первинний ключ у SQL Server автоматично налаштовується як стовпець IDENTITY.

## Явне налаштування генерації значень

EF Core автоматично налаштовує генерацію значень для первинних ключів, але ми можемо зробити те саме для неключових властивостей. Ви можете налаштувати будь-яку властивість так, щоб її значення генерувалося для вставлених сутностей наступним чином:

```cs
public class Blog
{
    public int BlogId { get; set; }
    public string Url { get; set; }

    [DatabaseGenerated(DatabaseGeneratedOption.Identity)]
    public DateTime Inserted { get; set; }
}
```
Аналогічно, властивість можна налаштувати так, щоб її значення генерувалося під час додавання або оновлення:

```cs
public class Blog
{
    public int BlogId { get; set; }
    public string Url { get; set; }

    [DatabaseGenerated(DatabaseGeneratedOption.Computed)]
    public DateTime LastUpdated { get; set; }
}
```

На відміну від значень за замовчуванням або обчислюваних стовпців, ми не вказуємо, як мають генеруватися значення; це залежить від використовуваного постачальника бази даних. Постачальники баз даних можуть автоматично налаштовувати генерацію значення для деяких типів властивостей, але інші можуть вимагати від вас ручного налаштування способу генерації значення.

Наприклад, на SQL Server, коли властивість GUID налаштовано як первинний ключ, постачальник автоматично виконує генерацію значень на стороні клієнта, використовуючи алгоритм для створення оптимальних послідовних значень GUID. Однак, якщо вказати ValueGeneratedOnAdd для властивості DateTime, це не матиме жодного ефекту (див. далі).

Аналогічно, властивості byte[], налаштовані як такі, що генеруються під час додавання або оновлення та позначені як токени паралельності, налаштовуються з типом даних rowversion, щоб значення автоматично генерувалися в базі даних. Однак, зазначення ValueGeneratedOnAdd не має жодного ефекту.

Зверніться до документації вашого постачальника, щоб дізнатися про конкретні методи генерації значень, які він підтримує.

## Генерація значень дати/часу

Поширеним запитом є наявність стовпця бази даних, який містить дату/час першої вставки рядка (значення, згенероване під час додавання) або останнього оновлення (значення, згенероване під час додавання або оновлення). Оскільки для цього існують різні стратегії, постачальники EF Core зазвичай не налаштовують автоматичне створення значень для стовпців дати/часу — вам доведеться налаштувати це самостійно.

### Мітка часу створення (timestamp)

Налаштування стовпця дати/часу для встановлення мітки часу створення рядка зазвичай полягає в налаштуванні значення за замовчуванням за допомогою відповідної функції SQL. Наприклад, на SQL Server можна використовувати наступне:

```cs
modelBuilder.Entity<Blog>()
    .Property(b => b.Created)
    .HasDefaultValueSql("getdate()");
```

Обов’язково виберіть відповідну функцію, оскільки їх може існувати кілька (наприклад, GETDATE() проти GETUTCDATE()).

### Оновлення позначки часу

Хоча збережені обчислені стовпці здаються гарним рішенням для керування позначками часу останнього оновлення, бази даних зазвичай не дозволяють вказувати такі функції, як GETDATE(), у обчислюваному стовпці. Як альтернативу, ви можете налаштувати тригер бази даних для досягнення того ж ефекту:

```sql
CREATE TRIGGER [dbo].[Blogs_UPDATE] ON [dbo].[Blogs]
    AFTER UPDATE
AS
BEGIN
    SET NOCOUNT ON;

    IF ((SELECT TRIGGER_NESTLEVEL()) > 1) RETURN;

    UPDATE B
    SET LastUpdated = GETDATE()
    FROM dbo.Blogs AS B
    INNER JOIN INSERTED AS I
        ON B.BlogId = I.BlogId
END
```

## Перевизначення генерації значень

Хоча властивість налаштована для генерації значень, у багатьох випадках ви все ще можете явно вказати для неї значення. Чи справді це працюватиме, залежить від конкретного налаштованого механізму генерації значень; хоча ви можете вказати явне значення замість використання значення стовпця за замовчуванням, те саме не можна зробити з обчислюваними стовпцями. Щоб перезаписати генерацію значення явним значенням, просто встановіть для властивості будь-яке значення, яке не є значенням CLR за замовчуванням для цього типу властивості (null для рядка, 0 для цілого числа, Guid.Empty для Guid тощо).

Щоб надати явне значення для властивостей, які були налаштовані як значення, згенеровані під час додавання або оновлення, необхідно також налаштувати властивість наступним чином:

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Blog>().Property(b => b.LastUpdated)
        .ValueGeneratedOnAddOrUpdate()
        .Metadata.SetAfterSaveBehavior(PropertySaveBehavior.Save);
}
```

## Немає генерації значень 

Окрім певних сценаріїв, таких як описані вище, для властивостей зазвичай не налаштовано генерацію значень; це означає, що застосунок завжди повинен надавати значення для збереження в базі даних. Це значення має бути призначене новим сутностям, перш ніж вони будуть додані до контексту.

Однак, у деяких випадках може знадобитися вимкнути генерацію значень, яка була налаштована за домовленістю. Наприклад, первинний ключ типу int зазвичай неявно налаштовується як значення, згенероване при додаванні (наприклад, стовпець ідентифікації на SQL Server). Ви можете вимкнути це за допомогою наступного:

```cs
public class Blog
{
    [DatabaseGenerated(DatabaseGeneratedOption.None)]
    public int BlogId { get; set; }

    public string Url { get; set; }
}
```
