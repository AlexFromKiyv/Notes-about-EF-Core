# Загальний огляд

## Зв'язки в об'єктних моделях

Зв'язок визначає, як дві сутності пов'язані одна з одною. Наприклад, під час моделювання публікацій(post) у блозі(blog) кожна публікація пов’язана з блогом, у якому вона опублікована, а блог пов’язаний з усіма публікаціями, опублікованими в цьому блозі.

В об'єктно-орієнтованій мові програмування, такій як C#, блог та публікація зазвичай представлені двома класами: Blog та Post. Наприклад:

```cs
public class Blog
{
    public string Name { get; set; }
    public virtual Uri SiteUri { get; set; }
}
```
```cs
public class Post
{
    public string Title { get; set; }
    public string Content { get; set; }
    public DateTime PublishedOn { get; set; }
    public bool Archived { get; set; }
}
```
У наведених вище класах немає нічого, що вказувало б на те, що Blog та Post пов'язані. Це можна додати до об'єктної моделі, додавши посилання з Post на Blog, на якому він опублікований:

```cs
public class Post
{
    public string Title { get; set; }
    public string Content { get; set; }
    public DateOnly PublishedOn { get; set; }
    public bool Archived { get; set; }

    public Blog Blog { get; set; } // Added
}
```
Аналогічно, протилежний напрямок того ж зв'язку можна представити як колекцію об'єктів Post у кожному Blog:

```cs
public class Blog
{
    public string Name { get; set; }
    public virtual Uri SiteUri { get; set; }

    public ICollection<Post> Posts { get; } // Added
}
```
Це з’єднання від блогу до публікації та, навпаки, від публікації до блогу називається «зв’язком» в EF Core.

Один зв'язок зазвичай можна переміщати в будь-якому напрямку. У цьому прикладі це від Blog до Post через властивість Blog.Posts, і від Post назад до Blog через властивість Post.Blog. Це один зв'язок, а не два.

У EF Core властивості Blog.Posts та Post.Blog називаються «навігаціями».

## Зв'язки в реляційних базах даних

Реляційні бази даних представляють зв'язки за допомогою зовнішніх ключів. Наприклад, за допомогою SQL Server або Azure SQL, для представлення наших класів Post та Blog можна використовувати такі таблиці:

```sql
CREATE TABLE [Posts] (
    [Id] int NOT NULL IDENTITY,
    [Title] nvarchar(max) NULL,
    [Content] nvarchar(max) NULL,
    [PublishedOn] datetime2 NOT NULL,
    [Archived] bit NOT NULL,
    [BlogId] int NOT NULL,
    CONSTRAINT [PK_Posts] PRIMARY KEY ([Id]),
    CONSTRAINT [FK_Posts_Blogs_BlogId] FOREIGN KEY ([BlogId]) REFERENCES [Blogs] ([Id]) ON DELETE CASCADE);

CREATE TABLE [Blogs] (
    [Id] int NOT NULL IDENTITY,
    [Name] nvarchar(max) NULL,
    [SiteUri] nvarchar(max) NULL,
    CONSTRAINT [PK_Blogs] PRIMARY KEY ([Id]));
```
У цій реляційній моделі таблицям «Post» та «Blog» надається стовпець «PRIMARY KEY». Значення первинного ключа однозначно ідентифікує кожну публікацію або блог. Крім того, таблиці «Post» надано стовпець «FOREIGN KEY». На стовпець первинного ключа Blogs з ідентифікатором посилається стовпець зовнішнього ключа BlogId таблиці Posts. Цей стовпець «обмежений» таким чином, що будь-яке значення у стовпці BlogId таблиці «Post» має збігатися зі значенням у стовпці Id таблиці «Blog». Це збігання визначає, з яким блогом пов’язана кожна публікація. Наприклад, якщо значення BlogId в одному рядку таблиці Posts дорівнює 7, то публікація, представлена ​​цим рядком, публікується в блозі з первинним ключем 7.

## Зіставлення зв'язків в EF Core

Зіставлення зв'язків в EF Core полягає у зіставленні представлення первинного/зовнішнього ключа, що використовується в реляційній базі даних, з посиланнями між об'єктами, що використовуються в об'єктній моделі.

У найпростішому сенсі це включає:
* Додавання властивості первинного ключа до кожного типу сутності.
* Додавання властивості зовнішнього ключа до одного типу сутності.
* Зв'язування посилань між типами сутностей з первинними та зовнішніми ключами для формування єдиної конфігурації зв'язку.

Після створення цього зіставлення, EF змінює значення зовнішніх ключів за потреби, коли змінюються посилання між об'єктами, та змінює посилання між об'єктами за потреби, коли змінюються значення зовнішніх ключів.

Наприклад, типи сутностей, показані вище, можна оновити за допомогою властивостей первинного та зовнішнього ключа:

```cs
public class Blog
{
    public int Id { get; set; }
    public string Name { get; set; }
    public virtual Uri SiteUri { get; set; }

    public ICollection<Post> Posts { get; }
}
```
```cs
public class Post
{
    public int Id { get; set; }
    public string Title { get; set; }
    public string Content { get; set; }
    public DateTime PublishedOn { get; set; }
    public bool Archived { get; set; }

    public int BlogId { get; set; }
    public Blog Blog { get; set; }
}
```
Властивості первинного та зовнішнього ключа не обов'язково мають бути публічно видимими властивостями типу сутності. Однак, навіть коли властивості приховані, важливо розуміти, що вони все ще існують у моделі EF.

Властивість первинного ключа Blog, Blog.Id, та властивість зовнішнього ключа Post, Post.BlogId, потім можуть бути пов'язані з посиланнями ("навігаціями") між типами сутностей (Blog.Posts та Post.Blog). Це робиться автоматично EF під час побудови такого простого зв'язку, але також може бути вказано явно під час перевизначення методу OnModelCreating вашого DbContext. Наприклад:

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Blog>()
        .HasMany(e => e.Posts)
        .WithOne(e => e.Blog)
        .HasForeignKey(e => e.BlogId)
        .HasPrincipalKey(e => e.Id);
}
```
Тепер усі ці властивості будуть працювати узгоджено разом як представлення єдиного зв'язку між блогом та публікацією.

Далі будуть розлянуті різні типи зв'язків. 

Моделі EF створюються за допомогою комбінації трьох механізмів: конвенцій, атрибутів відображення та API побудовника моделей. API побудови моделей є остаточним джерелом достовірної інформації для моделі EF – він завжди має пріоритет над конфігурацією, виявленою за домовленістю або визначеною шляхом зіставлення атрибутів. Це також єдиний механізм з повною точністю для налаштування кожного аспекту моделі EF.

## Використання зв'язків

Зв'язки, визначені в моделі, можна використовувати різними способами.

* Зв'язки можна використовувати для запиту пов'язаних даних будь-яким із трьох способів:
    * Оперативно, як частина запиту LINQ, використовуючи Include.
    * Ліниво, використовуючи проксі-сервери з лінивим завантаженням або лінивим завантаженням без проксі-серверів.
    * Явно, використовуючи методи Load або LoadAsync.

* Зв'язки можна використовувати для заповнення даних шляхом зіставлення значень PK зі значеннями FK.
* Зв'язки можна використовувати для відстеження графів сутностей. Потім відстежувач змін використовує зв'язки для:
    * Виявлення змін у зв'язках та їх виправлення
    * Виявлення змін у зв'язках та їх виправлення
Надсилання оновлень зовнішнього ключа до бази даних за допомогою SaveChanges або SaveChangesAsync
