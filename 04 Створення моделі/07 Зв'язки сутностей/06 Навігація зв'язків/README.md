# Навігація зв'язків

Зв'язки EF Core визначаються зовнішніми ключами. Навігація нашаровується поверх зовнішніх ключів, щоб забезпечити природний, об'єктно-орієнтований вигляд для читання та маніпулювання зв'язками. Використовуючи навігацію, програми можуть працювати з графами сутностей, не турбуючись про те, що відбувається зі значеннями зовнішніх ключів.

Кілька зв'язків не можуть спільно використовувати навігацію. Будь-який зовнішній ключ може бути пов'язаний максимум з однією навігацією від принципала до залежного та максимум з однією навігацією від залежного до принципала.

Немає потреби робити навігацію віртуальною, хіба що вона використовується проксі-серверами з відкладеним завантаженням або відстеженням змін.

## Навігація за посиланнями

Навігація буває двох форм: посилання та колекція. Посилання на навігацію – це прості посилання на об'єкти, що стосуються іншої сутності. Вони представляють сторону (сторони) "one" у зв'язках "one-to-many" та "one-to-one".

```cs
public Blog TheBlog { get; set; }
```
Навігація за посиланнями повинна мати сеттер, хоча він не обов'язково має бути публічним. Навігація за посиланнями не повинна автоматично ініціалізуватися ненульовим значенням за замовчуванням; це еквівалентно твердженню про існування сутності, коли її немає.

Під час використання типів посилань у C#, що допускають значення null, навігація посилань має бути null-допустимою для необов'язкових зв'язків:

```cs
public Blog? TheBlog { get; set; }
```
Навігація по посиланнях для обов'язкових зв'язків може бути як з можливістю додавання значення null, так і без нього.

## Навігація колекцій

Навігація колекцій є екземплярами типу колекції .NET, тобто будь-якого типу, що реалізує ICollection\<T\>. Колекція містить екземпляри пов'язаного типу сутності, яких може бути будь-яка кількість. Вони представляють сторону (сторони) "many" зв'язків "one-to-many" та "many-to-many".

```cs
public ICollection<Post> ThePosts { get; set; }
```
Навігація колекціями не потребує сеттера. Зазвичай колекцію ініціалізують вбудовано, тим самим усуваючи необхідність перевіряти, чи властивість має значення null.

```cs
public ICollection<Post> ThePosts { get; } = new List<Post>();
```

Не створюйте випадково властивість з тілом виразу, таку як:

```cs
public ICollection<Post> ThePosts => new List<Post>();

```
Це створюватиме новий, порожній екземпляр колекції щоразу, коли до властивості звертатимуться, і тому буде марним для навігації.

## Типи колекцій

Базовий екземпляр колекції повинен реалізовувати ICollection\<T\> та мати робочий метод Add. Зазвичай використовується List\<T\> або HashSet\<T\>. List\<T\> ефективний для невеликої кількості пов'язаних сутностей та підтримує стабільний порядок. HashSet\<T\> має ефективніший пошук для великої кількості сутностей, але не має стабільного впорядкування. Ви також можете використовувати власну реалізацію колекції.

Колекція повинна використовувати рівність посилань. Під час створення HashSet<T> для навігації колекцією обов'язково використовуйте ReferenceEqualityComparer.

Масиви не можна використовувати для навігації по колекціях, оскільки, навіть якщо вони реалізують ICollection\<T\>, метод Add викидає виняток під час виклику.

Навіть якщо екземпляр колекції має бути ICollection\<T\>, колекцію не потрібно відкривати як таку. Наприклад, поширеною є навігація як IEnumerable\<T\>, що забезпечує перегляд лише для читання, який не може бути випадковим чином змінений кодом програми.

```cs
public class Blog
{
    public int Id { get; set; }
    public IEnumerable<Post> ThePosts { get; } = new List<Post>();
}
```

Варіація цього шаблону включає методи для маніпулювання колекцією за потреби.

```cs
public class Blog
{
    private readonly List<Post> _posts = new();

    public int Id { get; set; }

    public IEnumerable<Post> Posts => _posts;

    public void AddPost(Post post) => _posts.Add(post);
}
```
Код застосунку все ще може перетворити відкриту колекцію на ICollection<T>, а потім маніпулювати нею. Якщо це викликає занепокоєння, то сутність може повернути захисну копію колекції.

```cs
public class Blog
{
    private readonly List<Post> _posts = new();

    public int Id { get; set; }

    public IEnumerable<Post> Posts => _posts.ToList();

    public void AddPost(Post post) => _posts.Add(post);
}
```
Ретельно подумайте, чи є отримана від цього цінність достатньо високою, щоб переважити накладні витрати на створення копії колекції щоразу, коли здійснюється доступ до навігації.

Цей останній шаблон працює, оскільки за замовчуванням EF отримує доступ до колекції через її поле резервного копіювання. Це означає, що EF сам додає та видаляє сутності з фактичної колекції, тоді як програми взаємодіють лише із захисною копією колекції.

## Ініціалізація навігації колекцій

Навігацію колекцій можна ініціалізувати типом сутності, або жадно:

```cs
public class Blog
{
    public ICollection<Post> Posts { get; } = new List<Post>();
}
```
Або ліниво:
```cs
public class Blog
{
    private ICollection<Post>? _posts;

    public ICollection<Post> Posts => _posts ??= new List<Post>();
}
```

Якщо EF потрібно додати сутність до навігації колекції, наприклад, під час виконання запиту, то він ініціалізує колекцію, якщо вона наразі має значення null. Створений екземпляр залежить від відкритого типу навігації.

* Якщо навігація відображається як HashSet\<T\>, тоді створюється екземпляр HashSet\<T\> за допомогою ReferenceEqualityComparer.
* В іншому випадку, якщо навігація надається як конкретний тип з конструктором без параметрів, тоді створюється екземпляр цього конкретного типу. Це стосується List\<T\>, але також і інших типів колекцій, включаючи користувацькі типи колекцій.
* В іншому випадку, якщо навігація відображається як IEnumerable\<T\>, ICollection\<T\> або ISet\<T\>, тоді створюється екземпляр HashSet\<T\> за допомогою ReferenceEqualityComparer.
* В іншому випадку, якщо навігація відображається як IList\<T\>, тоді створюється екземпляр List\<T\>.
* В іншому випадку виникає виняток.

##  Налаштування навігації

Навігація включається в модель як частина налаштування зв'язку. Тобто, за домовленістю, або за допомогою HasOne, HasMany тощо в API побудови моделі. Більшість налаштувань, пов'язаних з навігацією, виконується шляхом налаштування самого зв'язку. Однак, існують деякі типи конфігурації, які є специфічними для самих властивостей навігації, а не є частиною загальної конфігурації зв'язку. Цей тип конфігурації виконується за допомогою методу Navigation. Наприклад, щоб змусити EF отримувати доступ до навігації через його властивість, а не використовувати backing поле :

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Blog>()
        .Navigation(e => e.Posts)
        .UsePropertyAccessMode(PropertyAccessMode.Property);

    modelBuilder.Entity<Post>()
        .Navigation(e => e.Blog)
        .UsePropertyAccessMode(PropertyAccessMode.Property);
}
```

Виклик Navigation не можна використовувати для створення властивості навігації. Він використовується лише для налаштування властивості навігації, яка була створена раніше шляхом визначення зв'язку або з домовленості.

## Обов'язкові навігації

Навігація від залежного до принципала є обов'язковою, якщо зв'язок є обов'язковим, що, у свою чергу, означає, що властивість зовнішнього ключа не може мати значення null. І навпаки, навігація є необов'язковою, якщо зовнішній ключ може мати значення null, і тому зв'язок є необов'язковим.

Навігація за посиланнями від принципала до залежного є різною. У більшості випадків головна сутність завжди може існувати без будь-яких залежних сутностей. Тобто, обов'язковий зв'язок не означає, що завжди буде хоча б одна залежна сутність. У моделі EF, а також у реляційній базі даних немає способу гарантувати, що принципал пов'язаний з певною кількістю залежних. Якщо це необхідно, то це має бути реалізовано в логіці програми (бізнесу).

Існує один виняток із цього правила – коли головний та залежний типи використовують одну й ту саму таблицю в реляційній базі даних або містяться в документі. Це може статися з власними типами або невласними типами, що спільно використовують одну таблицю. У цьому випадку властивість навігації від принципала до залежного об'єкта може бути позначена як обов'язкова, що вказує на наявність залежного об'єкта. Налаштування властивості навігації за потреби виконується за допомогою методу Navigation.

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Blog>()
        .Navigation(e => e.BlogHeader)
        .IsRequired();
}
```
