# Домовленності для виявлення зв'язків.

EF Core використовує набір домовленостей під час виявлення та побудови моделі на основі класів типів сутностей.

Описані тут домовленості можна перевизначити шляхом явного налаштування зв'язку за допомогою атрибутів зіставлення або API побудови моделі.

## Виявлення навігацій

Виявлення зв'язків починається з виявлення навігацій між типами сутностей.

### Навігація за посиланнями

Властивість типу сутності виявляється як навігація за посиланнями, коли:
* Власність є public.
* Властивість має метод отримання та метод встановлення.
    * Метод сеттер не обов'язково має бути публічним; він може бути приватним або мати будь-який інший доступ.
    * Метод сеттер може бути доступним лише для ініціалізації(init-only).
* Тип властивості є або може бути типом сутності. Це означає, що тип
    * Повинен бути типом посилання.
    * Не повинен бути явно налаштований як примітивний тип властивості.
    * Не повинен бути відображений як примітивний тип властивості постачальником бази даних, що використовується.
    * Не повинен бути автоматично конвертованим у примітивний тип властивості, що відображається постачальником бази даних, що використовується.

* Властивість не є статичною.
* Властивість не є властивістю індексатора.

Наприклад, розглянемо такі типи сутностей:

```cs
public class Blog
{
    // Not discovered as reference navigations:
    public int Id { get; set; }
    public string Title { get; set; } = null!;
    public Uri? Uri { get; set; }
    public ConsoleKeyInfo ConsoleKeyInfo { get; set; }
    public Author DefaultAuthor => new() { Name = $"Author of the blog {Title}" };

    // Discovered as a reference navigation:
    public Author? Author { get; private set; }
}

public class Author
{
    // Not discovered as reference navigations:
    public Guid Id { get; set; }
    public string Name { get; set; } = null!;
    public int BlogId { get; set; }

    // Discovered as a reference navigation:
    public Blog Blog { get; init; } = null!;
}
```
Для цих типів Blog.Author та Author.Blog виявляються як навігаційні посилання. З іншого боку, такі властивості не виявляються як навігаційні посилання:
* Blog.Id, оскільки int є відображеним примітивним типом
* Blog.Title, оскільки string є відображеним примітивним типом
* Blog.Uri, оскільки Uri автоматично перетворюється на відображений примітивний тип
* Blog.ConsoleKeyInfo, оскільки ConsoleKeyInfo є типом значення C#
* Blog.DefaultAuthor, оскільки властивість не має сеттера
* Author.Id, оскільки Guid є відображеним примітивним типом
* Author.Name, оскільки string є відображеним примітивним типом
* Author.BlogId, оскільки int є відображеним примітивним типом 

### Навігація колекцій

Властивість типу сутності виявляється як навігація колекції, коли:
* Властивість є публічною.
* Властивість має метод геттер. Навігація колекцій може мати методи встановлення, але це не обов'язково.
* Тип властивості є або реалізує IEnumerable\<TEntity\>, де TEntity є або може бути типом сутності. Це означає, що тип TEntity:
    * Повинен бути типом посилання.
    * Не повинен бути явно налаштований як примітивний тип властивості.
    * Не повинен бути відображений як примітивний тип властивості постачальником бази даних, що використовується.
    * Не повинно бути автоматичним конвертуванням у примітивний тип властивості, що відображається використовуваним постачальником бази даних.
* Властивість не є статичною.
* Властивість не є властивістю індексатора.

Наприклад, у наступному коді як Blog.Tags, так і Tag.Blogs виявляються як навігації колекцій:

```cs
public class Blog
{
    public int Id { get; set; }
    public List<Tag> Tags { get; set; } = null!;
}

public class Tag
{
    public Guid Id { get; set; }
    public IEnumerable<Blog> Blogs { get; } = new List<Blog>();
}
```

### Сполучення навігацій

Після виявлення навігації, що переходить, наприклад, від типу сутності A до типу сутності B, необхідно визначити, чи має ця навігація інверсію, що йде у протилежному напрямку, тобто від типу сутності B до типу сутності A. Якщо така інверсія знайдена, то дві навігації сполучаються, утворюючи єдиний двонаправлений зв'язок. 

Тип зв'язку визначається тим, чи є навігація та її інверсія навігацією посиланням чи колекцією. Зокрема:

* Якщо одна навігація є навігацією колекції, а інша — навігацією посилання, то зв'язок є «one-to-many».
* Якщо обидві навігації є навігаціями за посиланнями, то зв'язок є «one-to-one».
* Якщо обидві навігації є навігаціями колекцій, то зв'язок є «many-to-many».

Виявлення кожного з цих типів зв'язків показано на прикладах нижче:

Зв'язок «one-to-many» між Blog та Post виявляється шляхом поєднання навігацій Blog.Posts та Post.Blog:

```cs
public class Blog
{
    public int Id { get; set; }
    public ICollection<Post> Posts { get; } = new List<Post>();
}

public class Post
{
    public int Id { get; set; }
    public int? BlogId { get; set; }
    public Blog? Blog { get; set; }
}
```

Єдиний зв'язок "one-to-one" між Blog та Author виявляється шляхом поєднання навігацій Blog.Author та Author.Blog:

```cs
public class Blog
{
    public int Id { get; set; }
    public Author? Author { get; set; }
}

public class Author
{
    public int Id { get; set; }
    public int? BlogId { get; set; }
    public Blog? Blog { get; set; }
}
```
Єдиний зв'язок «many-to-many» між Post та Tag виявляється шляхом поєднання навігацій Post.Tags та Tag.Posts:

```cs
public class Post
{
    public int Id { get; set; }
    public ICollection<Tag> Tags { get; } = new List<Tag>();
}

public class Tag
{
    public int Id { get; set; }
    public ICollection<Post> Posts { get; } = new List<Post>();
}
```
Таке сполучення навігацій може бути неправильним, якщо дві навігації представляють два різні односпрямовані зв'язки. У цьому випадку ці два зв'язки необхідно налаштувати явно.

Спарювання зв'язків працює лише тоді, коли між двома типами існує один зв'язок. Кілька зв'язків між двома типами необхідно налаштувати явно.

Описи тут наведено з точки зору зв'язків між двома різними типами. Однак, один і той самий тип може бути на обох кінцях зв'язку, і тому один тип може мати дві навігації, обидві пов'язані між собою. Це називається зв'язком самопосилання.

## Виявлення властивостей зовнішнього ключа

Після того, як навігації для зв'язку були виявлені або налаштовані явно, ці навігації використовуються для виявлення відповідних властивостей зовнішнього ключа для зв'язку. Після того, як навігації для зв'язку були виявлені або налаштовані явно, ці навігації використовуються для виявлення відповідних властивостей зовнішнього ключа для зв'язку.

* Тип властивості сумісний з первинним або альтернативним ключем основного типу сутності.
    * Типи сумісні, якщо вони однакові, або якщо тип властивості зовнішнього ключа є версією типу властивості первинного або альтернативного ключа, що може мати значення null.
* Назва властивості відповідає одній із домовленостей про іменування для властивості зовнішнього ключа. Домовленості про іменування такі:
    * \<navigation property name\>\<principal key property name\>
    * \<navigation property name>Id
    * \<principal entity type name\>\<principal key property name\>
    * \<principal entity type name\>Id

* Крім того, якщо залежний кінець був явно налаштований за допомогою API побудови моделі, і залежний первинний ключ сумісний, то залежний первинний ключ також буде використовуватися як зовнішній ключ.

Суфікс "Id" може мати будь-який регістр літер.

Наведені нижче типи сутностей наведено приклади для кожної з цих правил іменування.

Post.TheBlogKey виявляється як зовнішній ключ, оскільки він відповідає шаблону \<navigation property name\>\<principal key property name\>:

```cs
public class Blog
{
    public int Key { get; set; }
    public ICollection<Post> Posts { get; } = new List<Post>();
}

public class Post
{
    public int Id { get; set; }
    public int? TheBlogKey { get; set; }
    public Blog? TheBlog { get; set; }
}
```
Post.TheBlogID виявляється як зовнішній ключ, оскільки він відповідає шаблону \<navigation property name\>Id:

```cs
public class Blog
{
    public int Key { get; set; }
    public ICollection<Post> Posts { get; } = new List<Post>();
}

public class Post
{
    public int Id { get; set; }
    public int? TheBlogID { get; set; }
    public Blog? TheBlog { get; set; }
}
```
Post.BlogKey виявляється як зовнішній ключ, оскільки він відповідає шаблону \<principal entity type name\>\<principal key property name\>:

```cs
public class Blog
{
    public int Key { get; set; }
    public ICollection<Post> Posts { get; } = new List<Post>();
}

public class Post
{
    public int Id { get; set; }
    public int? BlogKey { get; set; }
    public Blog? TheBlog { get; set; }
}
```
Post.Blogid виявлено як зовнішній ключ, оскільки він відповідає шаблону \<principal entity type name\>Id:

```cs
public class Blog
{
    public int Key { get; set; }
    public ICollection<Post> Posts { get; } = new List<Post>();
}

public class Post
{
    public int Id { get; set; }
    public int? Blogid { get; set; }
    public Blog? TheBlog { get; set; }
}
```
У випадку навігації «один до багатьох», властивості зовнішнього ключа повинні бути типу з посиланням на навігацію, оскільки це буде залежна сутність. У випадку зв'язків «один до одного», виявлення властивості зовнішнього ключа використовується для визначення, який тип представляє залежний кінець зв'язку. Якщо властивість зовнішнього ключа не виявлена, то залежний кінець необхідно налаштувати за допомогою HasForeignKey.

Наведені вище правила також застосовуються до складених зовнішніх ключів, де кожна властивість складеного елемента повинна мати сумісний тип з відповідною властивістю первинного або альтернативного ключа, а кожна назва властивості повинна відповідати одній із описаних вище домовленостей щодо іменування.

## Визначення кардинальності

EF використовує виявлені навігації та властивості зовнішнього ключа для визначення кардинальності зв'язку разом з його головним та залежним кінцями:

* Якщо є одна непарна навігація за посиланнями, то зв'язок налаштовується як односпрямований «один до багатьох», з навігацією за посиланнями на залежному кінці.
* Якщо є одна непарна навігація колекції, то зв'язок налаштовується як односпрямований «один до багатьох», з навігацією колекції на головному кінці.
* Якщо є парні навігації по посиланнях та колекціях, то зв'язок налаштовується як двонаправлений «один до багатьох», з навігацією колекціями на головному кінці.
* Якщо навігація за посиланнями поєднана з іншою навігацією за посиланнями, тоді:
    * Якщо властивість зовнішнього ключа була виявлена ​​на одному боці, але не на іншому, то зв'язок налаштовується як двонаправлений один-до-одного, з властивістю зовнішнього ключа на залежному кінці.
    * В іншому випадку залежну сторону неможливо визначити, і EF викидає виняток, який вказує на те, що залежну сторону необхідно явно налаштувати.

* Якщо навігація колекції поєднана з іншою навігацією колекції, то зв'язок налаштовується як двонаправлений «багато до багатьох».

## Тіньові властивості зовнішнього ключа

Якщо EF визначив залежний кінець зв'язку, але не було виявлено жодної властивості зовнішнього ключа, тоді EF створить тіньову властивість для представлення зовнішнього ключа. Тіньова властивість:
* Має тип властивості первинного або альтернативного ключа на головному кінці зв'язку.
    * Тип за замовчуванням робить зв'язок необов'язковим за замовчуванням.
* Якщо на залежному кінці є навігація, то властивість тіньового зовнішнього ключа називається за допомогою цієї навігаційної назви, об'єднаної з назвою властивості первинного або альтернативного ключа.
* Якщо на залежному кінці немає навігації, то властивість тіньового зовнішнього ключа називається за допомогою імені типу сутності принципала, об'єднаного з іменем властивості первинного або альтернативного ключа.

## Каскадне видалення

За домовленістю, обов'язкові зв'язки налаштовані на каскадне видалення. Необов'язкові зв'язки налаштовані на відмову від каскадного видалення.

## Many-to-many

Зв'язки «many-to-many» не мають головного та залежного кінців, і жоден з кінців не містить властивості зовнішнього ключа. Натомість, зв'язки «many-to-many» використовують тип сутності об'єднання, який містить пари зовнішніх ключів, що вказують на будь-який кінець зв'язку «many-to-many». Розглянемо наступні типи сутностей, для яких зв'язок «many-to-many» виявляється за домовленістю:

```cs
public class Post
{
    public int Id { get; set; }
    public ICollection<Tag> Tags { get; } = new List<Tag>();
}

public class Tag
{
    public int Id { get; set; }
    public ICollection<Post> Posts { get; } = new List<Post>();
}
```
У цьому відкритті використовуються такі умовні позначення:
* Тип сутності об'єднання має назву <назва лівого типу сутності><назва правого типу сутності>. Отже, у цьому прикладі це PostTag.
    * Таблиця об'єднання має таку ж назву, як і тип сутності об'єднання.
* Типу сутності об'єднання надається властивість зовнішнього ключа для кожного напрямку зв'язку. Вони називаються <назва навігації><назва головного ключа>. Отже, у цьому прикладі властивості зовнішнього ключа - це PostsId та TagsId.
    * Для односпрямованого зв'язку "many-to-many" властивість зовнішнього ключа без пов'язаної навігації називається <назва типу головної сутності><назва головного ключа>.
* Властивості зовнішнього ключа не можуть мати значення null, що робить обидва зв'язки з об'єктом об'єднання обов'язковими. 
    * Домовленості про каскадне видалення означають, що ці зв'язки будуть налаштовані для каскадного видалення.
* Тип сутності об'єднання налаштовано зі складеним первинним ключем, що складається з двох властивостей зовнішнього ключа. Отже, у цьому прикладі первинний ключ складається з PostsId та TagsId.

## Індекси

За домовленістю, EF створює індекс бази даних для властивості або властивостей зовнішнього ключа. Тип створеного індексу визначається:

* Кардинальністю зв'язку
* Чи є зв'язок необов'язковим чи обов'язковим
* Кількістю властивостей, що складають зовнішній ключ

Для зв'язку «one-to-many» за домовленістю створюється простий індекс. Той самий індекс створюється для необов'язкових та обов'язкових зв'язків.

```sql
CREATE INDEX [IX_Post_BlogId] ON [Post] ([BlogId]);
```
Для обов'язкового зв'язку "one-to-one" створюється унікальний індекс.

```sql
CREATE UNIQUE INDEX [IX_Author_BlogId] ON [Author] ([BlogId]);
```
Для необов'язкових зв'язків "один до одного" на SQL Server додано фільтр IS NOT NULL для кращої обробки значень зовнішнього ключа Null.

```sql
CREATE UNIQUE INDEX [IX_Author_BlogId] ON [Author] ([BlogId]) WHERE [BlogId] IS NOT NULL;
```
Для складених зовнішніх ключів створюється індекс, що охоплює всі стовпці зовнішнього ключа.

```sql
CREATE INDEX "IX_Post_ContainingBlogId1_ContainingBlogId2" ON "Post" ("ContainingBlogId1", "ContainingBlogId2");
```
EF не створює індекси для властивостей, які вже охоплені існуючим індексом або обмеженням первинного ключа.

## Як зупинити створення індексів EF для зовнішніх ключів

Індекси мають додаткові витрати, і, як тут запитували, не завжди доречно створювати їх для всіх стовпців FK. Щоб досягти цього, ForeignKeyIndexConvention можна видалити під час побудови моделі:

```cs
protected override void ConfigureConventions(ModelConfigurationBuilder configurationBuilder)
{
    configurationBuilder.Conventions.Remove(typeof(ForeignKeyIndexConvention));
}
```
За потреби, індекси все ще можна явно створювати для тих стовпців зовнішнього ключа, які їх потребують.

## Назви обмежень зовнішнього ключа

За домовленістю обмеження зовнішнього ключа називаються FK_\<dependent type name\>_\<principal type name\>_\<foreign key property name\>. Для складених зовнішніх ключів \<foreign key property name\> стає списком назв властивостей зовнішнього ключа, розділених символами підкреслення.
