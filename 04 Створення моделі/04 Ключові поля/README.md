# Ключові поля

Ключ служить унікальним ідентифікатором для кожного екземпляра сутності. Більшість сутностей в EF мають один ключ, що відповідає концепції первинного ключа в реляційних базах даних. Сутності можуть мати додаткові ключі, окрім первинного ключа.

## Налаштування первинного ключа

За домовленістю, властивість з назвою Id або \<type name\>Id буде налаштована як первинний ключ сутності.

```cs
public class Car
{
    public string Id { get; set; }

    public string Make { get; set; }
    public string Model { get; set; }
}

public class Truck
{
    public string TruckId { get; set; }

    public string Make { get; set; }
    public string Model { get; set; }
}
```

Ви можете налаштувати окрему властивість як первинний ключ сутності наступним чином:

```cs
public class Car
{
    [Key]
    public string LicensePlate { get; set; }

    public string Make { get; set; }
    public string Model { get; set; }
}
```
Ви також можете налаштувати кілька властивостей як ключі сутності – це називається складеним ключем. Згідно з домовленостями, складений ключ встановлюватиметься лише в певних випадках, наприклад, для власної колекції типів.

```cs
[PrimaryKey(nameof(State), nameof(LicensePlate))]
public class Car
{
    public string State { get; set; }
    public string LicensePlate { get; set; }

    public string Make { get; set; }
    public string Model { get; set; }
}
```
## Генерація значень

Для нескладових числових первинних ключів та первинних ключів GUID EF Core налаштовує генерацію значень за домовленістю. Наприклад, числовий первинний ключ у SQL Server автоматично налаштовується як стовпець IDENTITY. Більш детально це питаня разгладається в окремомій главі далі.

## Ім'я первинного ключа

За домовленістю, у реляційних базах даних первинні ключі створюються з ім'ям PK_\<type name\>. Ви можете налаштувати ім'я обмеження первинного ключа наступним чином:

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Blog>()
        .HasKey(b => b.BlogId)
        .HasName("PrimaryKey_BlogId");
}
```

## Типи та значення ключів

Хоча EF Core підтримує використання властивостей будь-якого примітивного типу як первинного ключа, включаючи рядок, Guid, byte[] та інші, не всі бази даних підтримують усі типи як ключі. У деяких випадках значення ключів можна автоматично перетворити на підтримуваний тип, інакше перетворення слід вказати вручну.

Ключові властивості завжди повинні мати значення, відмінне від значення за замовчуванням, під час додавання нової сутності до контексту, але деякі типи будуть згенеровані базою даних. У такому випадку EF спробує згенерувати тимчасове значення, коли об'єкт буде додано для цілей відстеження. Після виклику SaveChanges тимчасове значення буде замінено значенням, згенерованим базою даних.

Якщо значення ключової властивості згенеровано базою даних, і під час додавання сутності вказано значення, відмінне від стандартного, тоді EF вважатиме, що сутність вже існує в базі даних, і спробує оновити її замість вставки нової. Щоб уникнути цього, вимкніть генерацію значень або перегляньте, як вказати явні значення для згенерованих властивостей.

## Альтернативні ключі

Альтернативний ключ служить альтернативним унікальним ідентифікатором для кожного екземпляра сутності на додаток до первинного ключа; його можна використовувати як ціль зв'язку. Під час використання реляційної бази даних це відповідає концепції унікального індексу/обмеження для стовпця(ів) альтернативного ключа та одного або кількох обмежень зовнішнього ключа, які посилаються на стовпець(и).

Якщо ви просто хочете забезпечити унікальність стовпця, визначте унікальний індекс(див. далі), а не альтернативний ключ. В EF альтернативні ключі доступні лише для читання та забезпечують додаткову семантику над унікальними індексами, оскільки їх можна використовувати як ціль для зовнішнього ключа.

Альтернативні ключі зазвичай вводяться за потреби, і вам не потрібно налаштовувати їх вручну. За домовленістю, альтернативний ключ вводиться, коли ви визначаєте властивість, яка не є первинним ключем, як ціль зв'язку.

```cs
internal class MyContext : DbContext
{
    public DbSet<Blog> Blogs { get; set; }
    public DbSet<Post> Posts { get; set; }

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Post>()
            .HasOne(p => p.Blog)
            .WithMany(b => b.Posts)
            .HasForeignKey(p => p.BlogUrl)
            .HasPrincipalKey(b => b.Url);
    }
}

public class Blog
{
    public int BlogId { get; set; }
    public string Url { get; set; }

    public List<Post> Posts { get; set; }
}

public class Post
{
    public int PostId { get; set; }
    public string Title { get; set; }
    public string Content { get; set; }

    public string BlogUrl { get; set; }
    public Blog Blog { get; set; }
}
```
Ви також можете налаштувати окрему властивість як альтернативний ключ:

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Car>()
        .HasAlternateKey(c => c.LicensePlate);
}
```
