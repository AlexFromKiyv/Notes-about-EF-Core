# Властивості сутності

Кожен тип сутності у вашій моделі має набір властивостей, які EF Core зчитуватиме та записуватиме з бази даних. Якщо ви використовуєте реляційну базу даних, властивості сутності зіставляються зі стовпцями таблиці.

## Включені та виключені властивості

За домовленістю, всі публічні властивості з методами getter та setter будуть включені до моделі. Певні властивості можна виключити наступним чином:

```cs
public class Blog
{
    public int BlogId { get; set; }
    public string Url { get; set; }

    [NotMapped]
    public DateTime LoadedFromDatabase { get; set; }
}

protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Blog>()
        .Ignore(b => b.LoadedFromDatabase);
}
```

## Назви стовпців

За домовленістю, під час використання реляційної бази даних властивості сутностей зіставляються зі стовпцями таблиці, які мають таку саму назву, як і властивість. Якщо ви бажаєте налаштувати стовпці з різними іменами, ви можете зробити це за допомогою наступного фрагмента коду:

```cs
public class Blog
{
    [Column("blog_id")]
    public int BlogId { get; set; }

    public string Url { get; set; }
}
```

## Типи даних стовпців

Під час використання реляційної бази даних постачальник бази даних вибирає тип даних на основі типу властивості .NET. Він також враховує інші метадані, такі як налаштована максимальна довжина, чи є властивість частиною первинного ключа тощо. 

Наприклад, постачальник SQL Server зіставляє властивості DateTime зі стовпцями datetime2(7), а рядкові властивості зі стовпцями nvarchar(max) (або з nvarchar(450) для властивостей, що використовуються як ключ).

Ви також можете налаштувати стовпці, щоб вказати точний тип даних для стовпця. Наприклад, наступний код налаштовує URL-адресу як рядок, що є non-unicode, з максимальною довжиною 200 та рейтинг як decimal з точністю 5 та шкалою 2:

```cs
public class Blog
{
    public int BlogId { get; set; }

    [Column(TypeName = "varchar(200)")]
    public string Url { get; set; }

    [Column(TypeName = "decimal(5, 2)")]
    public decimal Rating { get; set; }
}
```

## Максимальна довжина

Налаштування максимальної довжини надає підказку постачальнику бази даних щодо відповідного типу даних стовпця, який слід вибрати для заданої властивості. Максимальна довжина застосовується лише до типів даних масиву, таких як string та byte[].

У наступному прикладі налаштування максимальної довжини 500 призведе до створення стовпця типу nvarchar(500) на SQL Server:

```cs
public class Blog
{
    public int BlogId { get; set; }

    [MaxLength(500)]
    public string Url { get; set; }
}
```

### Точність і шкала

Деякі реляційні типи даних підтримують аспекти точності та масштабу; вони контролюють, які значення можна зберігати та скільки місця потрібно для стовпця. Які типи даних підтримують точність та масштаб, залежить від бази даних, але в більшості баз даних типи decimal та DateTime підтримують ці аспекти. Для властивостей decimal, precision визначає максимальну кількість цифр, необхідну для вираження будь-якого значення, яке містить стовпець, а scale визначає максимальну кількість знаків після коми. Для властивостей DateTime точність визначає максимальну кількість цифр, необхідну для вираження часток секунд, а масштаб не використовується.

Entity Framework does not do any validation of precision or scale before passing data to the provider. It is up to the provider or data store to validate as appropriate.
350
Entity Framework не виконує жодної перевірки точності чи масштабу перед передачею даних постачальнику. Постачальник або сховище даних повинні самостійно перевіряти відповідні значення. Наприклад, якщо цільовою аудиторією є SQL Server, стовпець типу даних datetime не дозволяє встановити точність, тоді як стовпець типу datetime2 може мати точність від 0 до 7 включно.

У наступному прикладі налаштування властивості Score з точністю 14 та масштабом 2 призведе до створення стовпця типу decimal(14,2) на SQL Server, а налаштування властивості LastUpdated з точністю 3 призведе до створення стовпця типу datetime2(3):

```cs
public class Blog
{
    public int BlogId { get; set; }
    [Precision(14, 2)]
    public decimal Score { get; set; }
    [Precision(3)]
    public DateTime LastUpdated { get; set; }
}
```

### Юнікод

Наприклад, у SQL Server nvarchar(x) використовується для представлення даних Unicode в UTF-16, тоді як varchar(x) використовується для представлення даних, відмінних від Unicode (але див. примітки щодо підтримки SQL Server UTF-8). Для баз даних, які не підтримують цю концепцію, це налаштування не має жодного ефекту. Властивості тексту за замовчуванням налаштовані як Unicode. Ви можете налаштувати стовпець як не-Unicode наступним чином:

```cs
public class Book
{
    public int Id { get; set; }
    public string Title { get; set; }

    [Unicode(false)]
    [MaxLength(22)]
    public string Isbn { get; set; }
}
```

## Обов'язкові та необов'язкові властивості

Властивість вважається необов'язковою, якщо вона може містити значення null. Якщо значення null не є допустимим для присвоєння властивості, то вона вважається обов'язковою властивістю. Під час зіставлення зі схемою реляційної бази даних обов'язкові властивості створюються як стовпці, що не допускають значення null, а необов'язкові властивості створюються як стовпці, що допускають значення null.

### Домовленість

За домовленістю, властивість, тип .NET якої може містити значення null, буде налаштована як необов'язкова, тоді як властивості, тип .NET якої не може містити значення null, будуть налаштовані як обов'язкові. Наприклад, усі властивості з типами значень .NET (int, decimal, bool тощо) налаштовуються як обов'язкові, а всі властивості з типами значень .NET, що можуть мати значення null (int?, decimal?, bool? тощо), налаштовуються як необов'язкові.

Типи посилань C#, що допускають значення null ( nullable reference types NRT), дозволяють анотувати типи посилань, вказуючи, чи допустимо для них містити значення null. Ця функція ввімкнена за замовчуванням у нових шаблонах проектів, але залишається вимкненою в існуючих проектах, якщо вона явно не ввімкнена. Типи посилань, що допускають значення Null, впливають на поведінку EF Core таким чином: 

    Якщо типи посилань, що допускають значення null, вимкнено, усі властивості з типами посилань .NET налаштовуються як необов'язкові за домовленістю (наприклад, рядок).

    Якщо ввімкнено типи посилань, що допускають значення null, властивості будуть налаштовані на основі можливості використання значення null у C# їхнього типу .NET: string? буде налаштовано як необов'язковий, але string буде налаштовано як обов'язковий.

У наступному прикладі показано тип сутності з обов'язковими та необов'язковими властивостями, з вимкненою та ввімкненою функцією посилання, що допускає значення null:

```cs

// With NRT
public class Customer
{
    public int Id { get; set; }
    public string FirstName { get; set; } // Required by convention
    public string LastName { get; set; } // Required by convention
    public string? MiddleName { get; set; } // Optional by convention

    // Note the following use of constructor binding, which avoids compiled warnings
    // for uninitialized non-nullable properties.
    public Customer(string firstName, string lastName, string? middleName = null)
    {
        FirstName = firstName;
        LastName = lastName;
        MiddleName = middleName;
    }
}

//Without NRT

public class CustomerWithoutNullableReferenceTypes
{
    public int Id { get; set; }

    [Required] // Data annotations needed to configure as required
    public string FirstName { get; set; }

    [Required] // Data annotations needed to configure as required
    public string LastName { get; set; }

    public string MiddleName { get; set; } // Optional by convention
}
```
Рекомендується використовувати типи посилань, що допускають значення null, оскільки це передає значення null, виражене в коді C#, до моделі EF Core та до бази даних, а також усуває необхідність використання Fluent API або анотацій даних для подвійного вираження тієї ж концепції.

### Явна конфігурація

Властивість, яка за домовленістю є необов'язковою, можна налаштувати як обов'язкову наступним чином:

```cs
public class Blog
{
    public int BlogId { get; set; }

    [Required]
    public string Url { get; set; }
}
```

## Сортування стовпців

Сортування можна визначити для текстових стовпців, визначаючи, як вони порівнюються та впорядковуються. Наприклад, наступний фрагмент коду налаштовує стовпець SQL Server без урахування регістру:

```cs
modelBuilder.Entity<Customer>().Property(c => c.Name)
    .UseCollation("SQL_Latin1_General_CP1_CI_AS");
```
Якщо всі стовпці в базі даних потребують певного сортування, визначте сортування на рівні бази даних.

## Коментарі до стовпця

Ви можете встановити довільний текстовий коментар, який буде встановлено до стовпця бази даних, що дозволить вам задокументувати вашу схему в базі даних:

```cs
public class Blog
{
    public int BlogId { get; set; }

    [Comment("The URL of the blog")]
    public string Url { get; set; }
}
```

## Порядок стовпців

За замовчуванням під час створення таблиці за допомогою міграцій EF Core впорядковує спочатку стовпці первинного ключа, потім властивості типу сутності та owned типів, і, нарешті, властивості базових типів. Однак ви можете вказати інший порядок стовпців:

```cs
public class EntityBase
{
    [Column(Order = 0)]
    public int Id { get; set; }
}

public class PersonBase : EntityBase
{
    [Column(Order = 1)]
    public string FirstName { get; set; }

    [Column(Order = 2)]
    public string LastName { get; set; }
}

public class Employee : PersonBase
{
    public string Department { get; set; }
    public decimal AnnualSalary { get; set; }
}
```
Зверніть увагу, що загалом більшість баз даних підтримують упорядкування стовпців лише під час створення таблиці. Це означає, що атрибут column order не можна використовувати для зміни порядку стовпців у існуючій таблиці.
