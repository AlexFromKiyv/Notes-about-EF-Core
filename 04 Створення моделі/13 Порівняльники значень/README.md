# Порівняльники значень

##  Передумови

Відстеження змін означає, що EF Core автоматично визначає, які зміни були внесені програмою до завантаженого екземпляра сутності, щоб ці зміни можна було зберегти назад у базу даних під час виклику SaveChanges. EF Core зазвичай виконує це, роблячи знімок екземпляра під час його завантаження з бази даних та порівнюючи цей знімок з екземпляром, переданим програмі. 

EF Core має вбудовану логіку для створення знімків та порівняння більшості стандартних типів, що використовуються в базах даних, тому користувачам зазвичай не потрібно турбуватися про це. Однак, коли властивість зіставляється через конвертер значень, EF Core має виконувати порівняння довільних типів користувачів, що може бути комплексно складним. За замовчуванням EF Core використовує порівняння рівності за замовчуванням, визначене типами (наприклад, метод Equals); для створення знімків копіюються типи значень для створення знімка, тоді як для посилальних типів копіювання не відбувається, і той самий екземпляр використовується як знімок.

У випадках, коли вбудована поведінка порівняння не підходить, користувачі можуть надати порівняльник значень, який містить логіку для створення знімків, порівняння та обчислення хеш-коду. Наприклад, наступний код налаштовує перетворення значення для властивості List\<int\>, яке має бути перетворене на рядок JSON у базі даних, а також визначає відповідний порівняльник значень:

```cs
modelBuilder
    .Entity<EntityType>()
    .Property(e => e.MyListProperty)
    .HasConversion(
        v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
        v => JsonSerializer.Deserialize<List<int>>(v, (JsonSerializerOptions)null),
        new ValueComparer<List<int>>(
            (c1, c2) => c1.SequenceEqual(c2),
            c => c.Aggregate(0, (a, v) => HashCode.Combine(a, v.GetHashCode())),
            c => c.ToList()));
```
Деталі прикладу розглядаються далі.

Зверніть увагу, що порівняльники значень також використовуються для визначення того, чи є два ключові значення однаковими під час розв'язання зв'язків; це пояснюється нижче.

## Поверхневе чи глибоке порівняння

Для малих, незмінних типів значень, таких як int, логіка EF Core за замовчуванням працює добре: значення копіюється як є під час створення знімка та порівнюється з вбудованим порівнянням рівності типу. Під час впровадження власного порівняльника значень важливо враховувати, чи доречна логіка глибокого чи поверхневого порівняння (і створення знімків).

Розглянемо масиви байтів, які можуть бути довільно великими. Їх можна порівняти:

* За посиланням, таким чином, що різниця виявляється лише у випадку використання нового байтового масиву
* Шляхом глибокого порівняння, таким чином, щоб виявилася мутація байтів у масиві

За замовчуванням EF Core використовує перший із цих підходів для неключових байтових масивів. Тобто, порівнюються лише посилання, а зміна виявляється лише тоді, коли існуючий байтовий масив замінюється новим. Це прагматичне рішення, яке дозволяє уникнути копіювання цілих масивів та їх побайтового порівняння під час виконання SaveChanges. Це означає, що поширений сценарій заміни, скажімо, одного зображення іншим обробляється продуктивним способом.

З іншого боку, рівність посилань не працюватиме, коли байтові масиви використовуються для представлення двійкових ключів, оскільки дуже малоймовірно, що властивість FK встановлена ​​на той самий екземпляр, що й властивість PK, з якою її потрібно порівняти. Таким чином, EF Core використовує глибокі порівняння для байтових масивів, що діють як ключі; це навряд чи призведе до значного зниження продуктивності, оскільки бінарні ключі зазвичай короткі. 

Зверніть увагу, що обрана логіка порівняння та створення знімків має відповідати одна одній: глибоке порівняння вимагає глибокого створення знімків для правильного функціонування.

## Прості незмінні класи

Розглянемо властивість, яка використовує конвертер значень для відображення простого незмінного класу.

```cs
public sealed class ImmutableClass
{
    public ImmutableClass(int value)
    {
        Value = value;
    }

    public int Value { get; }

    private bool Equals(ImmutableClass other)
        => Value == other.Value;

    public override bool Equals(object obj)
        => ReferenceEquals(this, obj) || obj is ImmutableClass other && Equals(other);

    public override int GetHashCode()
        => Value.GetHashCode();
}
```
```cs
modelBuilder
    .Entity<MyEntityType>()
    .Property(e => e.MyProperty)
    .HasConversion(
        v => v.Value,
        v => new ImmutableClass(v));
```

Властивості цього типу не потребують спеціальних порівнянь або знімків, оскільки:

* Рівність перевизначається, щоб різні екземпляри порівнювалися коректно.
* Тип незмінний, тому немає жодної можливості змінити значення знімка.

Отже, у цьому випадку поведінка EF Core за замовчуванням є нормальною.

## Прості незмінні структури

Зіставлення простих структур також просте та не вимагає спеціальних порівнянь чи створення знімків.

```cs
public readonly struct ImmutableStruct
{
    public ImmutableStruct(int value)
    {
        Value = value;
    }

    public int Value { get; }
}
```
```cs
modelBuilder
    .Entity<EntityType>()
    .Property(e => e.MyProperty)
    .HasConversion(
        v => v.Value,
        v => new ImmutableStruct(v));
```
EF Core має вбудовану підтримку для створення скомпільованих почленних порівнянь властивостей структур. Це означає, що для структур не потрібно перевизначати рівність для EF Core, але ви все одно можете зробити це з інших причин. Крім того, спеціальне створення знімків не потрібне, оскільки структури є незмінними та завжди копіюються почленно. (Це також стосується змінних структур, але змінних структур загалом слід уникати.)

## Змінні класи

Рекомендується використовувати незмінні типи (класи або структури) з перетворювачами значень, коли це можливо. Зазвичай це ефективніше та має чіткішу семантику, ніж використання змінного типу. Однак, з огляду на це, поширеною є використання властивостей типів, які застосунок не може змінити. Наприклад, відображення властивості, що містить список чисел:

```cs
public List<int> MyListProperty { get; set; }
```

Клас List\<T\>:

* Має рівність посилань; два списки, що містять однакові значення, розглядаються як різні.
* Є змінним; значення у списку можна додавати та видаляти.

Типове перетворення значень властивості списку може конвертувати список у JSON та з JSON:

```cs
modelBuilder
    .Entity<EntityType>()
    .Property(e => e.MyListProperty)
    .HasConversion(
        v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
        v => JsonSerializer.Deserialize<List<int>>(v, (JsonSerializerOptions)null));

var valueComparer = new ValueComparer<List<int>>(
    (c1, c2) => c1.SequenceEqual(c2),
    c => c.Aggregate(0, (a, v) => HashCode.Combine(a, v.GetHashCode())),
    c => c.ToList());

modelBuilder
    .Entity<EntityType>()
    .Property(e => e.MyListProperty)
    .Metadata
    .SetValueComparer(valueComparer);
```
Конструктор ValueComparer\<T\> приймає три вирази:

* Вираз для перевірки рівності
* Вираз для генерації хеш-коду
* Вираз для створення знімка значення

У цьому випадку порівняння виконується шляхом перевірки, чи послідовності чисел однакові.

Аналогічно, хеш-код будується з цієї ж послідовності. (Зауважте, що це хеш-код для змінних значень, і тому може спричинити проблеми. За можливості використовуйте незмінний варіант.)

Знімок створюється шляхом клонування списку за допомогою ToList. Знову ж таки, це потрібно лише тоді, коли списки будуть змінені. Натомість, якщо можливо, використовуйте незмінний тип.

Перетворювачі значень та порівняльники створюються за допомогою виразів, а не простих делегатів. Це пояснюється тим, що EF Core вставляє ці вирази в набагато складніше дерево виразів, яке потім компілюється в делегат формувача сутностей. Концептуально це схоже на вбудовування компілятора. Наприклад, просте перетворення може бути просто скомпільованим приведенням типів, а не викликом іншого методу для виконання перетворення.

## Порівняння ключів

Переконайтеся, що ви створили порівняльник, який підходить для ключів, коли встановлюєте його для властивості первинного, основного або зовнішнього ключа. Використовуйте SetKeyValueComparer у рідкісних випадках, коли для однієї властивості потрібна різна семантика.

## Перевизначення порівняльника за замовчуванням

Іноді порівняння за замовчуванням, яке використовується EF Core, може бути невідповідним. Наприклад, мутація байтових масивів за замовчуванням не виявляється в EF Core. Це можна перевизначити, встановивши інший порівняльник для властивості:

```cs
modelBuilder
    .Entity<EntityType>()
    .Property(e => e.MyBytes)
    .Metadata
    .SetValueComparer(
        new ValueComparer<byte[]>(
            (c1, c2) => c1.SequenceEqual(c2),
            c => c.Aggregate(0, (a, v) => HashCode.Combine(a, v.GetHashCode())),
            c => c.ToArray()));
```

EF Core тепер порівнюватиме послідовності байтів і, таким чином, виявлятиме мутації масиву байтів.
