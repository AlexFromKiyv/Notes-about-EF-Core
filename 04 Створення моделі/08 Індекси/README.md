# Індекси

Індекси – це поширена концепція в багатьох сховищах даних. Хоча їхня реалізація у сховищі даних може відрізнятися, вони використовуються для підвищення ефективності пошуку на основі стовпця (або набору стовпців).

Ви можете вказати індекс для стовпця наступним чином

```cs
[Index(nameof(Url))]
public class Blog
{
    public int BlogId { get; set; }
    public string Url { get; set; }
}
```
За домовленістю, індекс створюється в кожній властивості (або наборі властивостей), які використовуються як зовнішній ключ.

## Складений індекс

Індекс також може охоплювати більше одного стовпця:

```cs
[Index(nameof(FirstName), nameof(LastName))]
public class Person
{
    public int PersonId { get; set; }
    public string FirstName { get; set; }
    public string LastName { get; set; }
}
```
Індекси за кількома стовпцями, також відомі як складені індекси, пришвидшують запити, які фільтрують за стовпцями індексу, а також запити, які фільтрують лише за першими стовпцями, що охоплюються індексом.

## Унікальність індексу

За замовчуванням індекси не є унікальними: кілька рядків можуть мати однакові значення для набору стовпців індексу. Ви можете зробити індекс унікальним наступним чином:

```cs
[Index(nameof(Url), IsUnique = true)]
public class Blog
{
    public int BlogId { get; set; }
    public string Url { get; set; }
}
```
Спроба вставити більше однієї сутності з однаковими значеннями для набору стовпців індексу призведе до виникнення виняткової ситуації.

## Порядок сортування індексу

У більшості баз даних кожен стовпець, що охоплюється індексом, може бути як за зростанням, так і за спаданням. Для індексів, що охоплюють лише один стовпець, це зазвичай не має значення: база даних може проходити по індексу у зворотному порядку за потреби. Однак, для складених індексів порядок може бути вирішальним для гарної продуктивності та може означати різницю між тим, чи буде індекс використаний запитом, чи ні. Загалом, порядок сортування стовпців індексу повинен відповідати порядку, зазначеному в реченні ORDER BY вашого запиту.

Порядок сортування індексу за замовчуванням – зростання. Ви можете налаштувати сортування всіх стовпців у порядку спадання наступним чином:

```cs
[Index(nameof(Url), nameof(Rating), AllDescending = true)]
public class Blog
{
    public int BlogId { get; set; }
    public string Url { get; set; }
    public int Rating { get; set; }
}
```
Ви також можете вказати порядок сортування для кожного стовпця наступним чином:

```cs
[Index(nameof(Url), nameof(Rating), IsDescending = new[] { false, true })]
public class Blog
{
    public int BlogId { get; set; }
    public string Url { get; set; }
    public int Rating { get; set; }
}
```

## Найменування індексів та множинні індекси

За домовленістю, індекси, створені в реляційній базі даних, називаються IX_\<type name\>_\<property name\>.
Для складених індексів \<property name\> стає списком імен властивостей, розділених символами підкреслення.

Ви можете встановити ім'я індексу, створеного в базі даних:

```cs
[Index(nameof(Url), Name = "Index_Url")]
public class Blog
{
    public int BlogId { get; set; }
    public string Url { get; set; }
}
```
Зверніть увагу, що якщо ви викликаєте HasIndex більше одного разу для одного й того ж набору властивостей, це продовжує налаштовувати один індекс, а не створює новий:

```cs
modelBuilder.Entity<Person>()
    .HasIndex(p => new { p.FirstName, p.LastName })
    .HasDatabaseName("IX_Names_Ascending");

modelBuilder.Entity<Person>()
    .HasIndex(p => new { p.FirstName, p.LastName })
    .HasDatabaseName("IX_Names_Descending")
    .IsDescending();
```
Оскільки другий виклик HasIndex перевизначає перший, створюється лише один спадний індекс. Це може бути корисним для подальшого налаштування індексу, створеного за домовленістю.

```cs
modelBuilder.Entity<Person>()
    .HasIndex(p => new { p.FirstName, p.LastName }, "IX_Names_Ascending");

modelBuilder.Entity<Person>()
    .HasIndex(p => new { p.FirstName, p.LastName }, "IX_Names_Descending")
    .IsDescending();
```
Зверніть увагу, що це ім'я також використовується як ім'я за замовчуванням для імені бази даних, тому явний виклик HasDatabaseName не потрібен.

## Фільтр індексу

Деякі реляційні бази даних дозволяють вказувати фільтрований або частковий індекс. Це дозволяє індексувати лише підмножину значень стовпця, зменшуючи розмір індексу та покращуючи як продуктивність, так і використання дискового простору.

Ви можете використовувати Fluent API для визначення фільтра для індексу, наданого як SQL-вираз.

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Blog>()
        .HasIndex(b => b.Url)
        .HasFilter("[Url] IS NOT NULL");
}
```
Під час використання постачальника SQL Server, EF додає фільтр «IS NOT NULL» для всіх стовпців, що допускають значення null і є частиною унікального індексу. Щоб змінити цю домовленість, можна вказати значення null.

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Blog>()
        .HasIndex(b => b.Url)
        .IsUnique()
        .HasFilter(null);
}
```

## Включені стовпці

Деякі реляційні бази даних дозволяють налаштувати набір стовпців, які включаються до індексу, але не є частиною його «ключа». Це може значно покращити продуктивність запитів, коли всі стовпці запиту включені до індексу як ключові або неключові стовпці, оскільки до самої таблиці не потрібно звертатися. 

У наступному прикладі стовпець URL є частиною ключа індексу, тому будь-яка фільтрація запитів за цим стовпцем може використовувати індекс. Крім того, запити, які звертаються лише до стовпців Title та PublishedOn, не потребуватимуть доступу до таблиці та виконуватимуться ефективніше:

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<Post>()
        .HasIndex(p => p.Url)
        .IncludeProperties(
            p => new { p.Title, p.PublishedOn });
}
```

## Перевірка обмежень

Перевірка обмежень – це стандартна реляційна функція, яка дозволяє визначити умову, яка має виконуватися для всіх рядків таблиці; будь-яка спроба вставити або змінити дані, що порушують обмеження, буде невдалою. Обмеження перевірки подібні до обмежень, що не є null (які забороняють null-значення у стовпці), або до обмежень унікальності (які забороняють дублікати), але дозволяють визначати довільні SQL-вирази.

Ви можете використовувати Fluent API для визначення обмеження перевірки для таблиці, наданого як SQL-вираз:

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder
        .Entity<Product>()
        .ToTable(b => b.HasCheckConstraint("CK_Prices", "[Price] > [DiscountedPrice]"));
}
```
В одній таблиці можна визначити кілька обмежень перевірки, кожне з яких має власну назву. Примітка: деякі поширені обмеження перевірки можна налаштувати за допомогою спільного пакета EFCore.CheckConstraints.
