# 02 Клієнт або сервер для обчислень.

Як правило, Entity Framework Core намагається максимально точно обчислити запит на сервері. EF Core перетворює частини запиту на параметри, які він може обчислити на стороні клієнта. Решта запиту (разом зі згенерованими параметрами) передається постачальнику бази даних для визначення еквівалентного запиту до бази даних для обчислень на сервері.

EF Core підтримує часткове обчислення на клієнті в проекції верхнього рівня (по суті, останній виклик Select()). Якщо проекцію верхнього рівня в запиті неможливо перевести на сервер, EF Core отримає всі необхідні дані з сервера та обчислює решту частин запиту на клієнті. 
Якщо проекцію верхнього рівня в запиті неможливо перевести на сервер, EF Core отримає всі необхідні дані з сервера та обчислює решту частин запиту на клієнті.

## Обчислення на клієнті в проекції верхнього рівня.

У наступному прикладі допоміжний метод використовується для стандартизації URL-адрес блогів, які повертаються з бази даних SQL Server.

```cs
    public static string StandardizeUrl(string url)
    {
        url = url.ToLower();

        if (!url.StartsWith("https://"))
        {
            url = string.Concat("https://", url);
        }

        return url;
    }
```


Оскільки постачальник SQL Server не має уявлення про те, як реалізовано цей метод, неможливо перевести його в SQL. Усі інші аспекти запиту обчислюються в базі даних, але передача поверненої URL-адреси через цей метод виконується на клієнті.

```cs
    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        modelBuilder.Entity<Blog>()
        .HasData(
        new Blog { BlogId = 1, Url = @"https://devblogs.microsoft.com/dotnet", Rating = 5 },
        new Blog { BlogId = 2, Url = @"mytravelblog.com/", Rating = 4 });

    }
```

```cs
var blogs = await context.Blogs
    .OrderByDescending(blog => blog.Rating)
    .Select(
        blog => new { Id = blog.BlogId, Url = StandardizeUrl(blog.Url) })
    .ToListAsync();
```
```
1       https://devblogs.microsoft.com/dotnet
2       https://mytravelblog.com/
```

## Непідтримуване обчислення на клієнті

Хоча обчислення на клієнті корисне, іноді воно може призвести до низької продуктивності. Розглянемо наступний запит, у якому допоміжний метод тепер використовується у фільтрі where.

```cs
var blogs = await context.Blogs
    .Where(blog => StandardizeUrl(blog.Url).Contains("dotnet"))
    .ToListAsync();
```

Оскільки фільтр не можна застосувати в базі даних, усі дані потрібно витягнути в пам'ять, щоб застосувати фільтр на клієнті. Залежно від фільтра та обсягу даних на сервері, оцінка клієнта може призвести до низької продуктивності. Отже, Entity Framework Core блокує таку оцінку клієнта та викидає виняток виконання.

## Явнє обчислення на клієнті

У певних випадках, як-от у наведених нижче, може знадобитися примусово ввести обчислення на клієнті явно.

* Обсяг даних невеликий, тому обчислення на клієнті не призводить до значного зниження продуктивності. 
* Використаний оператор LINQ не має серверної трансляції.

У таких випадках ви можете явно ввімкнути обчислення на клієнті, викликавши методи, такі як AsEnumerable або ToList (AsAsyncEnumerable або ToListAsync для async). Використовуючи AsEnumerable, ви б передавали результати потоково, але використання ToList призведе до буферизації шляхом створення списку, що також займатиме додаткову пам'ять. Хоча, якщо ви перераховуєте кілька разів, то зберігання результатів у списку допомагає більше, оскільки запит до бази даних лише один. Залежно від конкретного використання, слід оцінити, який метод є більш корисним для даного випадку.

## Потенційний витік пам'яті під час обчислення на клієнті

Оскільки переклад та компіляція запитів є дорогими, EF Core кешує скомпільований план запиту. Кешований делегат може використовувати клієнтський код під час виконання обчислення на клієнті проекції верхнього рівня. EF Core генерує параметри для частин дерева, що обчислення на клієнті, і повторно використовує план запиту, замінюючи значення параметрів. Але певні константи у дереві виразів не можна перетворити на параметри. Якщо кешований делегат містить такі константи, то ці об'єкти не можуть бути вилучені зі сміття, оскільки на них все ще посилаються. Якщо такий об'єкт містить DbContext або інші служби, це може призвести до зростання використання пам'яті програмою з часом. Така поведінка зазвичай є ознакою витоку пам'яті. EF Core викидає виняток щоразу, коли зустрічає константи типу, який неможливо зіставити за допомогою поточного постачальника бази даних. Типові причини та їх вирішення наведені нижче:

* Використання методу екземпляра: Під час використання методів екземпляра в клієнтській проекції дерево виразів містить константу екземпляра. Якщо ваш метод не використовує жодних даних з екземпляра, розгляньте можливість зробити метод статичним. Якщо вам потрібні дані екземпляра в тілі методу, передайте ці конкретні дані як аргумент методу.

* Передача константних аргументів методу: Цей випадок зазвичай виникає при використанні this в аргументі клієнтського методу. Розглянте розділення аргументу на кілька скалярних аргументів, які можуть бути зіставлені постачальником бази даних.

* Інші константи: Якщо константа зустрічається в будь-якому іншому випадку, ви можете оцінити, чи потрібна ця константа в обробці. Якщо константа необхідна або якщо ви не можете використати рішення з наведених вище випадків, створіть локальну змінну для зберігання значення та використовуйте локальну змінну в запиті. EF Core перетворить локальну змінну на параметр.
