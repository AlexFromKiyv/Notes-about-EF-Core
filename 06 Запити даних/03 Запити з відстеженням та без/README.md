# 03 Запити з відстеженням та без

Поведінка відстеження контролює, чи зберігає Entity Framework Core інформацію про екземпляр сутності у своєму трекері змін. Якщо сутність відстежується, будь-які виявлені в ній зміни зберігаються в базі даних під час SaveChanges. EF Core також виправляє властивості навігації між сутностями в результаті запиту відстеження та сутностями, що знаходяться в системі відстеження змін.

Типи сутностей без ключа ніколи не відстежуються. Скрізь, де в цій статті згадуються типи сутностей, маються на увазі типи сутностей, у яких визначено ключ.

## Відстежувані запити

За замовчуванням запити, що повертають типи сутностей, є відстежувальними. Відстежувані запити означає, що будь-які зміни екземплярів сутностей зберігаються за допомогою SaveChanges. У наступному прикладі зміна рейтингу блогу виявляється та зберігається в базі даних під час SaveChanges:

```cs
var blog = await context.Blogs.SingleOrDefaultAsync(b => b.BlogId == 1);
blog.Rating = 5;
await context.SaveChangesAsync();
```

Коли результати повертаються в відстежувані запиті, EF Core перевіряє, чи сутність вже знаходиться в контексті. Якщо EF Core знаходить існуючу сутність, повертається той самий екземпляр, що потенційно може використовувати менше пам'яті та бути швидшим, ніж запит без відстеження. EF Core не перезаписує поточні та початкові значення властивостей сутності в записі значеннями бази даних. Якщо сутність не знайдено в контексті, EF Core створює новий екземпляр сутності та приєднує його до контексту. Результати запиту не містять жодної сутності, яка додана до контексту, але ще не збережена в базі даних.

## Запити без відстеження

Запити без відстеження корисні, коли результати використовуються в сценарії лише для читання. Зазвичай вони виконуються швидше, оскільки немає потреби налаштовувати інформацію для відстеження змін. Якщо об'єкти, отримані з бази даних, не потребують оновлення, слід використовувати запит без відстеження. Окремий запит можна налаштувати як такий, що не відстежується. Запит без відстеження також видає результати на основі даних бази даних, не враховуючи будь-які локальні зміни чи додані об'єкти.

```cs
var blogs = await context.Blogs
    .AsNoTracking()
    .ToListAsync();
```
Поведінку відстеження за замовчуванням можна змінити на рівні екземпляра контексту:

```cs
context.ChangeTracker.QueryTrackingBehavior = QueryTrackingBehavior.NoTracking;

var blogs = await context.Blogs.ToListAsync();
```
У наступному розділі пояснюється, коли запит без відстеження може бути менш ефективним, ніж запит із відстеженням.

## Розв'язання ідентифікації

Оскільки відстежувальні запити використовує засіб відстеження змін, EF Core виконує розв'язання ідентифікаційних даних у відстежувальних запитах. Під час матеріалізації сутності EF Core повертає той самий екземпляр сутності з трекера змін, якщо вона вже відстежується. Якщо результат містить ту саму сутність кілька разів, для кожного виникнення повертається той самий екземпляр. Запити без відстеження:

* Не використовуе трекер змін і не виконуе ідентифікацію результату.
* Повертає новий екземпляр сутності, навіть якщо та сама сутність міститься в результаті кілька разів.

Відстеження та відсутність відстеження можна поєднувати в одному запиті. Тобто, ви можете мати запит без відстеження, який виконує розв'язання ідентифікаційних даних у результатах. Так само, як і для оператора запиту AsNoTracking, ми додали ще один оператор AsNoTrackingWithIdentityResolution\<TEntity\>(IQueryable\<TEntity\>). Також до переліку QueryTrackingBehavior додано пов'язаний запис. Коли запит для використання розпізнавання ідентифікаторів налаштовано без відстеження, окремий трекер змін використовується у фоновому режимі під час генерації результатів запиту, тому кожен екземпляр матеріалізується лише один раз. Оскільки цей трекер змін відрізняється від того, що знаходиться в контексті, результати не відстежуються контекстом. Після повної нумерації запиту, система відстеження змін виходить з області видимості, а сміття збирається за потреби.

## Налаштування поведінки відстеження за замовчуванням

Якщо ви змінюєте поведінку відстеження для багатьох запитів, можливо, вам варто змінити натомість поведінку за замовчуванням:

```cs
protected override void OnConfiguring(DbContextOptionsBuilder optionsBuilder)
{
    optionsBuilder
        .UseSqlServer(@"Server=(localdb)\mssqllocaldb;Database=EFQuerying.Tracking;Trusted_Connection=True;ConnectRetryCount=0")
        .UseQueryTrackingBehavior(QueryTrackingBehavior.NoTracking);
}
```
Це робить усі ваші запити невідстежуваними за замовчуванням. Ви все ще можете додати AsTracking, щоб відстежувати певні запити.

## Відстеження та користувацьких проеуціях

Навіть якщо тип результату запиту не є типом сутності, EF Core все одно відстежуватиме типи сутностей, що містяться в результаті, за замовчуванням. У наступному запиті, який повертає анонімний тип, будуть відстежуватися екземпляри Blog у результуючому наборі.

```cs
var query = context.Blogs
    .Select(
        b =>
        new { Blog = b, PostCount = b.Posts.Count() });
```

Якщо результуючий набір містить типи сутностей, що походять з композиції LINQ, EF Core відстежуватиме їх.

```cs
var blog = context.Blogs
    .Select(
        b =>
            new { Blog = b, Post = b.Posts.OrderBy(p => p.Rating).LastOrDefault() });
```

Якщо результуючий набір не містить жодних типів сутностей, то відстеження не виконується. У наступному запиті ми повертаємо анонімний тип з деякими значеннями з сутності (але без екземплярів фактичного типу сутності). Із запиту не виведено жодних відстежуваних сутностей.

```cs
    var query = context.Blogs
        .Select(
            b =>
                new { Id = b.BlogId, b.Url });

    var blogs = await query.ToListAsync();

    foreach (var blog in blogs)
    {
        Console.WriteLine($"{blog.Id}\t{blog.Url}");
    }
```

EF Core підтримує обчислення на клієнті в проекції верхнього рівня. Якщо EF Core матеріалізує екземпляр сутності для оцінки клієнтом, він буде відстежуватися. Тут, оскільки ми передаємо сутності блогу клієнтському методу StandardizeURL, EF Core також відстежуватиме екземпляри блогу.

```cs
var blogs = await context.Blogs
    .OrderByDescending(blog => blog.Rating)
    .Select(
        blog => new { Id = blog.BlogId, Url = StandardizeUrl(blog) })
    .ToListAsync();
```
EF Core не відстежує екземпляри сутностей без ключів, що містяться в результаті. Але EF Core відстежує всі інші екземпляри типів сутностей з ключем відповідно до наведених вище правил.
