# Застосування міграцій

Після додавання міграцій їх потрібно розгорнути та застосувати до ваших баз даних. Існує кілька стратегій для цього, деякі з яких більше підходять для виробничого середовища, а інші — для життєвого циклу розробки.

Якою б не була ваша стратегія розгортання, завжди перевіряйте згенеровані міграції та тестуйте їх перед застосуванням до робочої бази даних. Міграція може видалити стовпець, хоча метою було його перейменування, або може завершитися невдачею з різних причин під час застосування до бази даних.

## SQL-скрипти

Рекомендований спосіб розгортання міграцій до робочої бази даних – це створення SQL-скриптів. Переваги цієї стратегії включають наступне:

* SQL-скрипти можна перевірити на точність; це важливо, оскільки застосування змін схеми до робочих баз даних є потенційно небезпечною операцією, яка може призвести до втрати даних.
* У деяких випадках скрипти можна налаштувати відповідно до конкретних потреб виробничої бази даних.
* SQL-скрипти можна використовувати разом із технологією розгортання, і навіть їх можна створювати як частину вашого процесу неперервної інтеграції (CI).
* Скрипти SQL можна надавати DBA, а також керувати ними та архівувати окремо. 

### Базове використання

Наступний код генерує SQL-скрипт з пустої бази даних до останньої міграції:

```
dotnet ef migrations script
```

### З заданої міграції

Наведений нижче код генерує SQL-скрипт із заданої міграції до останньої міграції.

```
dotnet ef migrations script 20260712094513_AddMakers
```

### З від та до

Наведений нижче код генерує SQL-скрипт із зазначеного `from migration` до зазначеного `to migration`.

```
dotnet ef migrations script AddNewTables AddAuditTable
```
Ви можете використовувати об'єкт from, який є новішим за об'єкт to, щоб згенерувати скрипт відкату.

Будь ласка, зверніть увагу на можливі сценарії втрати даних.

Генерація скрипта приймає такі два аргументи, щоб вказати, який діапазон міграцій слід згенерувати:

* Міграція з (від) має бути останньою міграцією, застосованою до бази даних перед запуском скрипта.Якщо міграції не застосовувалися, вкажіть 0 (це значення за замовчуванням).
* Міграція (до) – це остання міграція, яка буде застосована до бази даних після запуску скрипта. За замовчуванням це остання міграція у вашому проєкті. 

### Ідемпотентні SQL-скрипти

Згенеровані вище SQL-скрипти можна застосовувати лише для зміни схеми з однієї міграції на іншу; ви несете відповідальність за належне застосування скрипта та лише до баз даних у правильному стані міграції. EF Core також підтримує генерацію ідемпотентних скриптів, які внутрішньо перевіряють, які міграції вже застосовані (через таблицю історії міграцій), і застосовують лише відсутні. Це корисно, якщо ви точно не знаєте, яка була остання міграція, застосована до бази даних, або якщо ви розгортаєте на кількох базах даних, кожна з яких може перебувати на різних етапах міграції.

```
dotnet ef migrations script --idempotent
```

## Інструменти командного рядка

Інструменти командного рядка EF можна використовувати для застосування міграцій до бази даних. Хоча цей підхід продуктивний для локальної розробки та тестування міграцій, він не ідеально підходить для керування виробничими базами даних:

* Команди SQL застосовуються безпосередньо інструментом, не надаючи розробнику можливості їх перевірити або змінити. Це може бути небезпечно у виробничому середовищі.
* Пакет .NET SDK та інструмент EF мають бути встановлені на робочих серверах, а також для них потрібен вихідний код проекту.

Наступні дії оновлюють вашу базу даних до останньої міграції:

```
dotnet ef database update
```
Наступні дії оновлюють вашу базу даних відповідно до заданої міграції:

```
dotnet ef database update AddNewTables
```
Зверніть увагу, що це також можна використовувати для повернення до попередньої міграції. 

Будь ласка, зверніть увагу на можливі сценарії втрати даних.

Щоб отримати додаткові відомості про застосування міграцій за допомогою інструментів командного рядка, див. окремий розділ нижче.

## Пакети міграції

Пакети міграції – це виконувані файли з одного файлу, які можна використовувати для застосування міграцій до бази даних. Вони усувають деякі недоліки SQL-скриптів та інструментів командного рядка:

* Виконання SQL-скриптів вимагає додаткових інструментів.
* Обробка транзакцій та поведінка цих інструментів щодо продовження роботи при помилці є непослідовними, а іноді й неочікуваними. Це може призвести до того, що ваша база даних перестане бути визначена, якщо під час застосування міграцій станеться збій.
* Пакети можна створювати як частину вашого процесу неперервної інтеграції (CI) та легко виконувати пізніше в рамках процесу розгортання.
* Пакети можна виконувати без встановлення .NET SDK або EF Tool (або навіть .NET Runtime, якщо вони автономні), і вони не потребують вихідного коду проекту.

Наступний код генерує пакет:

```
dotnet ef migrations bundle
```
Наступний код генерує автономний пакет для Linux:

```
dotnet ef migrations bundle --self-contained -r linux-x64
```

### efbundle

Отриманий виконуваний файл за замовчуванням називається efbundle. Його можна використовувати для оновлення бази даних до останньої міграції. Це еквівалентно запуску dotnet ef database update або Update-Database.
Можливості запуску можна отритмати наступним викликом

```
.\efbundle.exe --help
```

У наступному прикладі застосовується міграція до локального екземпляра SQL Server з використанням вказаного імені користувача та облікових даних:

```
.\efbundle.exe --connection 'Data Source=(local)\MSSQLSERVER;Initial Catalog=Blogging;User ID=myUsername;Password={;'$Credential;'here'}'
```
Не забудьте скопіювати appsettings.json разом з вашим пакетом. Пакет залежить від наявності appsettings.json у каталозі виконання.

## Приклад пакета міграції

Після того, як міграції будуть готові до розгортання, створіть пакет за допомогою dotnet ef migrations.

```
dotnet ef migrations bundle
```
Вихід — це виконуваний файл, придатний для вашої цільової операційної системи. У моєму випадку це Windows x64, тому efbundle.exe знаходиться в моїй локальній папці. Запуск цього виконуваного файлу застосовує міграції, що містяться в ньому:

```
D:\Temp\Draft\Draft>.\efbundle.exe
```
Як і у випадку з dotnet ef database update, міграції застосовуються до бази даних, лише якщо вони ще не були застосовані. Наприклад, повторний запуск того самого пакета нічого не дасть, оскільки нових міграцій для застосування немає:

```
No migrations were applied. The database is already up to date.
Done.
```
Однак, якщо до моделі вносяться зміни та за допомогою dotnet ef migrations add генеруються додаткові міграції, їх можна об'єднати в новий виконуваний файл, готовий до застосування.

```
 dotnet ef migrations bundle --force
```
Опцію --force можна використовувати для перезапису існуючого пакета новим.

Виконання цього нового пакета застосовує нові міграції до бази даних:

```
Applying migration '20260713091557_ChangeMarekAddRating'.
```
Оновлена збірка виконає всі міграції якшо база даних не створена.


За замовчуванням пакет використовує рядок підключення до бази даних з конфігурації вашої програми. Однак, можна перенести іншу базу даних, передавши рядок підключення в командному рядку.

```
.\efbundle.exe --connection "Data Source=(LocalDb)\MSSQLLocalDB;Database=SixOhProduction"
```
Цього разу було застосовано всі міграції, оскільки жодна з них ще не була застосована до робочої бази даних.

## Застосування міграцій під час виконання

Сама програма може застосовувати міграції програмно, зазвичай під час запуску. Хоча цей підхід продуктивний для локальної розробки та тестування міграцій, він не підходить для керування виробничими базами даних з таких причин:

* Для версій EF до 9, якщо запущено кілька екземплярів вашої програми, обидві програми можуть спробувати застосувати міграцію одночасно та зазнати невдачі (або, що ще гірше, спричинити пошкодження даних).
* Аналогічно, якщо одна програма отримує доступ до бази даних, поки інша програма застосовує міграції, це може спричинити серйозні проблеми.
* Для зміни схеми бази даних програма повинна мати підвищені права доступу. Зазвичай гарною практикою є обмеження прав доступу до бази даних програми у робочому середовищі.
* Важливо мати можливість відкотити застосовану міграцію у разі виникнення проблеми. Інші стратегії забезпечують це легко та одразу після встановлення.
* Команди SQL застосовуються безпосередньо програмою, без можливості для розробника перевірити або змінити їх. Це може бути небезпечно у виробничому середовищі.

Щоб застосувати міграції програмно, викличте context.Database.MigrateAsync(). Наприклад, типова програма ASP.NET може виконувати такі дії:

```cs
public static async Task Main(string[] args)
{
    var host = CreateHostBuilder(args).Build();

    using (var scope = host.Services.CreateScope())
    {
        var db = scope.ServiceProvider.GetRequiredService<ApplicationDbContext>();
        await db.Database.MigrateAsync();
    }

    host.Run();
}
```
Зверніть увагу, що MigrateAsync() будується на основі сервісу IMigrator, який можна використовувати для складніших сценаріїв. Використовуйте myDbContext.GetInfrastructure().GetService<IMigrator>() для доступу до нього.

    * Ретельно обміркуйте, перш ніж використовувати цей підхід у робочому середовищі. Досвід показав, що простота цієї стратегії розгортання переважується проблемами, які вона створює. Натомість розгляньте можливість створення SQL-скриптів з міграцій.

    * Не викликайте EnsureCreatedAsync() перед MigrateAsync(). EnsureCreatedAsync() обходить міграції для створення схеми, що призводить до збою MigrateAsync().

## Блокування міграції

Починаючи з EF Core 9, MigrateAsync та Migrate автоматично отримують блокування для всієї бази даних перед застосуванням будь-яких міграцій. Це захищає від пошкодження бази даних, яке може виникнути внаслідок одночасного виконання міграцій кількома екземплярами програм, що є поширеним сценарієм під час застосування міграцій під час виконання. Блокування утримується протягом усього виконання міграції, включаючи будь-який код заповнення, і автоматично знімається після завершення операції.

Блокування міграції застосовується, коли міграції застосовуються за допомогою будь-якого з наведених нижче методів.

* dotnet ef database update (.NET CLI)
* Update-Database (Package Manager Console)
* Migration bundles
* MigrateAsync and Migrate (runtime migration)

SQL-скрипти не залежать від блокування міграції, оскільки вони застосовуються поза межами EF Core.

Починаючи з EF Core 9, виклик Migrate() або MigrateAsync() викличе виняток, коли модель має зміни, що очікують обробки, порівняно з останньою міграцією (ідентифікатор події попередження RelationalEventId.PendingModelChangesWarning). Щоб виявити цю умову перед розгортанням, скористайтеся командою dotnet ef migrations has-pending-model-changes у вашому конвеєрі CI/CD. За потреби попередження можна приховати за допомогою ConfigureWarnings (ігноруючи RelationalEventId.PendingModelChangesWarning), але зазвичай це не рекомендується у виробничих сценаріях. Див.

Механізм блокування суттєво відрізняється залежно від постачальника баз даних і може пов'язуватися з проблемами, характерними для кожного постачальника. Наприклад, постачальник SQLite використовує таблицю блокування, яка може бути загублена, якщо процес несподівано завершиться. Завжди звертайтеся до документації вашого постачальника для отримання детальної інформації.

## Обмеження

* Огортання MigrateAsync у явну транзакцію не підтримується.
* У SQLite покинуті блокування міграції можуть блокувати наступні міграції.
