# Керування міграціями

У міру змін вашої моделі міграції додаються та видаляються в рамках звичайної розробки, а файли міграції перевіряються в системі керування версіями вашого проекту. Щоб керувати міграціями, спочатку потрібно інсталювати засоби командного рядка EF Core.

Якщо DbContext знаходиться в іншій збірці, ніж проект запуску, ви можете явно вказати цільовий та проект запуску або в інструментах консолі диспетчера пакетів, або в інструментах .NET CLI.

## Додати міграцію

Після зміни моделі ви можете додати міграцію для цієї зміни:

```
dotnet ef migrations add AddBlogCreatedTimestamp
```
Ім’я міграції можна використовувати як повідомлення коміту в системі контролю версій. Наприклад, ви можете вибрати назву типу AddBlogCreatedTimestamp, якщо зміна є новою властивістю CreatedTimestamp у вашій сутності Blog.

До вашого проєкту в каталозі Migrations буде додано три файли:

* XXXXXXXXXXXXXX_AddBlogCreatedTimestamp.cs – основний файл міграцій. Містить операції, необхідні для застосування міграції (у Up) та для її скасування (у Down).
* XXXXXXXXXXXXXX_AddBlogCreatedTimestamp.Designer.cs – файл метаданих міграцій. Містить інформацію, що використовується EF.
* MyContextModelSnapshot.cs – знімок вашої поточної моделі. Використовується для визначення того, що змінилося під час додавання наступної міграції.

Мітка часу в назві файлу допомагає впорядкувати їх у хронологічному порядку, щоб ви могли бачити прогрес змін.

## Простори імен

Ви можете вільно переміщувати файли міграцій та змінювати їхній простір імен вручну. Нові міграції створюються як брати і сестри останньої міграції. Або ж ви можете вказати каталог під час генерації наступним чином:

```
dotnet ef migrations add InitialCreate --output-dir Your/Directory
```

## Налаштування коду міграції

Хоча EF Core зазвичай створює точні міграції, завжди слід перевіряти код і переконатися, що він відповідає бажаним змінам; у деяких випадках це навіть необхідно зробити.

### Перейменування стовпців

Одним із помітних прикладів, коли потрібне налаштування міграцій, є перейменування властивості. Наприклад, якщо перейменувати властивість з Name на FullName, EF Core згенерує таку міграцію:

```cs
migrationBuilder.DropColumn(
    name: "Name",
    table: "Customers");

migrationBuilder.AddColumn<string>(
    name: "FullName",
    table: "Customers",
    nullable: true);
```
EF Core зазвичай не може визначити, коли потрібно видалити стовпець і створити новий (дві окремі зміни), а коли стовпець слід перейменувати. Якщо вищезазначену міграцію застосувати як є, усі імена ваших клієнтів будуть втрачені. Щоб перейменувати стовпець, замініть вищезгадану згенеровану міграцію на наступне:

```cs
migrationBuilder.RenameColumn(
    name: "Name",
    table: "Customers",
    newName: "FullName");
```
Процес створення каркаса міграції попереджає, коли операція може призвести до втрати даних (наприклад, видалення стовпця). Якщо ви бачите це попередження, обов’язково перевірте код міграції на точність.

## Додавання необробленого SQL

Хоча перейменування стовпця можна здійснити за допомогою вбудованого API, у багатьох випадках це неможливо. Наприклад, ми можемо захотіти замінити існуючі властивості FirstName та LastName однією новою властивістю FullName. Міграція, згенерована EF Core, буде такою:

```cs
migrationBuilder.DropColumn(
    name: "FirstName",
    table: "Customer");

migrationBuilder.DropColumn(
    name: "LastName",
    table: "Customer");

migrationBuilder.AddColumn<string>(
    name: "FullName",
    table: "Customer",
    nullable: true);
```
Як і раніше, це призвело б до небажаної втрати даних. Щоб перенести дані зі старих стовпців, ми переставляємо міграції та вводимо необроблену SQL-операцію наступним чином:

```cs
        protected override void Up(MigrationBuilder migrationBuilder)
        {
            migrationBuilder.AddColumn<string>(
                name: "FullName",
                table: "Customers",
                nullable: true);

            migrationBuilder.Sql(
                @"
                    UPDATE Customers
                    SET FullName = FirstName + ' ' + LastName;
                ");

            migrationBuilder.DropColumn(
                name: "FirstName",
                table: "Customers");
            
            migrationBuilder.DropColumn(
                name: "LastName",
                table: "Customers");

        }
```

## Довільні зміни за допомогою необробленого SQL

Необроблений SQL також можна використовувати для керування об'єктами бази даних, про які EF Core не знає. Для цього додайте міграцію без внесення жодних змін до моделі; буде згенеровано пусту міграцію, яку потім можна заповнити необробленими операціями SQL.

Наприклад, наступна міграція створює збережену процедуру SQL Server:

```cs
        protected override void Up(MigrationBuilder migrationBuilder)
        {
            string sql = @"CREATE PROCEDURE dbo.GetCustomerFullNames
                            AS
                            BEGIN
                                SET NOCOUNT ON;

                                SELECT 
                                    Id,
                                    CONCAT(FirstName, CHAR(9), LastName) AS FullName
                                FROM 
                                    dbo.Customers;
                            END;";

            migrationBuilder.Sql(
                $" EXEC ('{sql}')");
        }
```

EXEC використовується, коли оператор має бути першим або єдиним у пакеті SQL. Його також можна використовувати для обходу помилок парсера в ідемпотентних скриптах міграції, які можуть виникати, коли стовпці, на які посилаються, наразі не існують у таблиці.

Це можна використовувати для керування будь-яким аспектом вашої бази даних, зокрема:

* Stored procedures
* Full-Text Search
* Functions
* Triggers
* Views

У більшості випадків EF Core автоматично обгортатиме кожну міграцію в окрему транзакцію під час застосування міграцій. На жаль, деякі операції міграції не можуть бути виконані в межах транзакції в деяких базах даних; у цих випадках ви можете відмовитися від транзакції, передавши suppressTransaction: true до migrationBuilder.Sql.

## Видалення міграції

Іноді ви додаєте міграцію та розумієте, що вам потрібно внести додаткові зміни до моделі EF Core перед її застосуванням. Щоб видалити останню міграцію, скористайтеся цією командою.

```
dotnet ef migrations remove
```
Після видалення міграції ви можете внести додаткові зміни до моделі та додати її знову.

Уникайте видалення будь-яких міграцій, які вже були застосовані до робочих баз даних. Це означає, що ви не зможете скасувати ці міграції з баз даних і можете порушити припущення, зроблені наступними міграціями.

## Список міграцій

Ви можете переглянути список усіх існуючих міграцій наступним чином:

```
dotnet ef migrations list
```

## Перевірка наявності змін моделі, що очікують на виконання

Іноді вам може знадобитися перевірити, чи були внесені якісь зміни до моделі з моменту останньої міграції. Це може допомогти вам дізнатися, коли ви або ваш колега забули додати міграцію. Один зі способів зробити це – скористатися цією командою.

```
dotnet ef migrations has-pending-model-changes
```
Ви також можете виконати цю перевірку програмно за допомогою context.Database.HasPendingModelChanges(). Це можна використовувати для написання модульного тесту, який завершується невдачею, якщо ви забудете додати міграцію.

## Скидання всіх міграцій

У деяких крайніх випадках може знадобитися видалити всі міграції та почати все спочатку. Це можна легко зробити, видаливши папку Migrations та базу даних; після цього ви можете створити нову початкову міграцію, яка міститиме всю вашу поточну схему.

Також можна скинути всі міграції та створити одну без втрати даних. Іноді це називається "стисканням" і передбачає певну ручну роботу:

1. Зробіть резервну копію бази даних, на випадок, якщо щось піде не так.
2. У вашій базі даних видаліть усі рядки з таблиці історії міграцій (наприклад, DELETE FROM [__EFMigrationsHistory] на SQL Server).
3. Видаліть папку Migrations.
4. Створіть нову міграцію та згенеруйте для неї SQL-скрипт (dotnet ef migrations script).
5. Вставте один рядок в історію міграцій, щоб записати, що першу міграцію вже застосовано, оскільки ваші таблиці вже там є. SQL-запит вставки – це остання операція у згенерованому вище SQL-скрипті, яка має такий вигляд (не забудьте оновити значення):

```sql
INSERT INTO [__EFMigrationsHistory] ([MIGRATIONID], [PRODUCTVERSION])
VALUES (N'<full_migration_timestamp_and_name>', N'<EF_version>');
```
Будь-який користувацький код міграції буде втрачено після видалення папки Migrations. Будь-які налаштування необхідно застосувати до нової початкової міграції вручну, щоб їх було збережено.
