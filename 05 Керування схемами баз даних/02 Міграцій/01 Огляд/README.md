# Огляд

У реальних проектах моделі даних змінюються в міру впровадження функцій: додаються та видаляються нові сутності або властивості, а схеми баз даних необхідно відповідно змінювати, щоб синхронізуватися з програмою. Функція міграції в EF Core надає спосіб поступового оновлення схеми бази даних, щоб синхронізувати її з моделлю даних програми, зберігаючи при цьому існуючі дані в базі даних.

На загальному рівні міграції функціонують наступним чином:

* Коли вноситься зміна в модель даних, розробник використовує інструменти EF Core для додавання відповідної міграції, що описує оновлення, необхідні для синхронізації схеми бази даних. EF Core порівнює поточну модель зі знімком старої моделі, щоб визначити відмінності, та генерує вихідні файли міграції; файли можна відстежувати в системі керування версіями вашого проекту, як і будь-який інший вихідний файл.

* Після створення нової міграції її можна застосувати до бази даних різними способами. EF Core записує всі застосовані міграції у спеціальну таблицю історії, що дозволяє йому знати, які міграції були застосовані, а які ні.

## Початок роботи

Припустимо, ви щойно завершили створення своєї першої програми EF Core, яка містить таку просту модель:

```cs
public class Blog
{
    public int Id { get; set; }
    public string Name { get; set; }
}
```

## Встановлення інструментів

Спочатку вам потрібно буде встановити інструменти командного рядка EF Core:

```
dotnet tool install --global dotnet-ef
```

## Створення вашої першої міграції

Тепер ви готові додати свою першу міграцію. Запустіть команду в каталозі проекту:

```
dotnet ef migrations add InitialCreate
```
EF Core створить у вашому проєкті каталог під назвою Migrations та згенерує деякі файли. Варто перевірити, що саме згенерував EF Core, і, можливо, внести зміни, але зараз ми це пропустимо.

## Створення бази даних та схеми

На цьому етапі ви можете дозволити EF створити базу даних та схему з міграції. Це можна зробити за допомогою наступного:

```
dotnet ef database update
```
Ось і все — ваш застосунок готовий до роботи на вашій новій базі даних, і вам не потрібно було писати жодного рядка SQL. Зауважте, що цей спосіб застосування міграцій ідеально підходить для локальної розробки, але менш підходить для виробничих середовищ.

## Розвиток вашої моделі

Минуло кілька днів, і вас попросили додати позначку часу створення до ваших блогів. Ви внесли необхідні зміни до своєї програми, і тепер ваша модель виглядає так:

```cs
public class Blog
{
    public int Id { get; set; }
    public string Name { get; set; }
    public DateTime CreatedTimestamp { get; set; }
}
```
Ваша модель та ваша виробнича база даних тепер не синхронізовані – нам потрібно додати новий стовпець до схеми вашої бази даних. Давайте створимо для цього нову міграцію:

```
dotnet ef migrations add AddBlogCreatedTimestamp
```
Зверніть увагу, що ми надаємо міграціям описові назви, щоб пізніше було легше зрозуміти історію проєкту.

Оскільки це не перша міграція проєкту, EF Core тепер порівнює вашу оновлену модель зі знімком старої моделі, до того, як стовпець було додано; знімок моделі – це один із файлів, згенерованих EF Core під час додавання міграції, і він реєструється в системі керування версіями. На основі цього порівняння EF Core виявляє, що стовпець було додано, і додає відповідну міграцію.

Тепер ви можете застосувати міграцію, як і раніше:

```
dotnet ef database update
```
Зверніть увагу, що цього разу EF виявляє, що база даних вже існує. Крім того, коли ми застосували нашу першу міграцію вище, цей факт був записаний у спеціальній таблиці історії міграцій у вашій базі даних; це дозволяє EF автоматично застосовувати лише нову міграцію.

## Виключення частин вашої моделі

Іноді вам може знадобитися посилатися на типи з іншого DbContext. Це може призвести до конфліктів міграції. Щоб запобігти цьому, виключіть цей тип з міграцій одного з DbContexts.

```cs
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    modelBuilder.Entity<IdentityUser>()
        .ToTable("AspNetUsers", t => t.ExcludeFromMigrations());
}
```
