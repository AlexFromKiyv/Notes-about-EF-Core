# Використання окремого проекту міграцій

Можливо, вам знадобиться зберігати міграції в іншому проекті, ніж той, що містить ваш DbContext. Це рекомендується, якщо ваш проект використовує тип проекту, специфічний для платформи, такий як WinUI, Xamarin, MAUI, Blazor або Azure Functions, або якщо він орієнтований на певний ідентифікатор середовища виконання (RID). Ви також можете використовувати цю стратегію для підтримки кількох наборів міграцій, наприклад, одного для розробки, а іншого для оновлень між випусками.

## Дії

1. Створіть нову бібліотеку класів.
2. Додайте посилання на ваш проект DbContext.
3. Перемістіть файли міграцій та знімків моделі до бібліотеки класів.

Якщо у вас немає існуючих міграцій, згенеруйте одну в проекті, що містить DbContext, а потім перемістіть її. Це важливо, оскільки якщо проект міграцій не містить існуючої міграції, команда Add-Migration не зможе знайти DbContext.

4. Налаштуйте збірку міграцій:

```cs
services.AddDbContext<ApplicationDbContext>(
    options =>
        options.UseSqlServer(
            Configuration.GetConnectionString("DefaultConnection"),
            x => x.MigrationsAssembly("WebApplication1.Migrations")));
```
5. Додайте посилання на ваш проект міграції зі стартового проекту.

Якщо це спричиняє циклічну залежність, ви можете оновити базовий вихідний шлях проекту міграції:


Якщо ви все зробили правильно, ви зможете додати нові міграції до проєкту.

```
dotnet ef migrations add NewMigration --project AppMigrations
```
