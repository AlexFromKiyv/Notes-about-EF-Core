# Шаблони зворотного проектування для користувача

Під час зворотного проектування Entity Framework Core прагне створювати каркас якісного коду загального призначення, який можна використовувати в різних типах програм, і використовує загальні правила кодування для узгодженого вигляду та звичного відчуття. Однак іноді бажаним є більш спеціалізований код та альтернативні стилі кодування.

У цій статті показано, як налаштувати риштований код за допомогою текстових шаблонів T4.

## Необхідні умови

У цій статті розраховано, що ви знайомі з реверсивним проектуванням в EF Core. Якщо ні, будь ласка, ознайомтеся з цією статтею, перш ніж продовжувати.

## Додавання шаблонів за замовчуванням

Першим кроком до налаштування коду каркасу є додавання шаблонів за замовчуванням до вашого проекту. Шаблони за замовчуванням – це ті, що використовуються EF Core внутрішньо під час зворотного проектування. Вони надають вам відправну точку для початку налаштування каркасу коду. Почніть з інсталяції пакета шаблонів EF Core для dotnet new:

```
dotnet new install Microsoft.EntityFrameworkCore.Templates@9.0.16
```
Тепер ви можете додати шаблони за замовчуванням до свого проєкту. Зробіть це, виконавши таку команду з каталогу вашого проєкту.

```
dotnet new ef-templates
```
Ця команда додає до вашого проєкту такі файли.

CodeTemplates/
* EFCore/
    * DbContext.t4
    * EntityType.t4

Шаблон DbContext.t4 використовується для створення каркасу класу DbContext для бази даних, а шаблон EntityType.t4 використовується для створення каркасу класів типів сутностей для кожної таблиці та представлення в базі даних. 

Розширення .t4 використовується (замість .tt), щоб запобігти трансформації шаблонів програмою Visual Studio. Натомість шаблони будуть трансформовані EF Core.

## Вступ до T4

Відкриємо шаблон DbContext.t4 та перевіримо його вміст. Цей файл є текстовим шаблоном T4. T4 — це мова для генерації тексту за допомогою .NET. Наведений нижче код наведено лише для ілюстрації; він не відображає повний вміст файлу.

Текстові шаблони T4, особливо ті, що генерують код, можуть бути важко читабельними без підсвічування синтаксису. За потреби знайдіть розширення для вашого редактора коду, яке вмикає підсвічування синтаксису T4.

```t4
<#@ template hostSpecific="true" #>
<#@ assembly name="Microsoft.EntityFrameworkCore.Design" #>
<#@ parameter name="NamespaceHint" type="System.String" #>
<#@ import namespace="Microsoft.EntityFrameworkCore" #>
<#
    if (!string.IsNullOrEmpty(NamespaceHint))
    
#>
namespace <#= NamespaceHint #>;
```

Перші кілька рядків, що починаються з <#@, називаються директивами. Вони впливають на те, як перетворюється шаблон. У наступній таблиці коротко описано кожен тип директив, що використовуються.


|Директива|Опис|
|---------|----|
|template|Визначає hostSpecific="true", що дозволяє використовувати властивість Host у шаблоні для доступу до служб EF Core.|
|assembly|Додає посилання на збірку, необхідні для компіляції шаблону.|
|parameter|Оголошує параметри, які будуть передані EF Core під час перетворення шаблону.|
|import|Подібно до using C#, вносить простори імен у область видимості коду шаблону..|

Після директив наступний розділ DbContext.t4 називається блоком керування. Стандартний блок керування починається з <# та закінчується #>. Код всередині нього буде виконано під час перетворення шаблону. Список властивостей та методів, доступних всередині блоків керування, див. у класі TextTransformation. 

Все, що знаходиться поза межами блоку керування, буде скопійовано безпосередньо у вивід шаблону.

Блок керування виразом починається з <#=. Код всередині нього буде обчислено, а результат буде додано до виводу шаблону. Це схоже на інтерпольовані рядкові аргументи C#.

Для більш детального та повного пояснення синтаксису T4 див. статтю Написання текстового шаблону T4.

## Налаштування типів сутностей

Давайте розглянемо, як налаштувати шаблон. За замовчуванням EF Core генерує наступний код для властивостей навігації колекції.

```cs
public virtual ICollection<Album> Albums { get; } = new List<Album>();
```
Використання List<T> є гарним значенням за замовчуванням для більшості програм. Однак, якщо ви використовуєте фреймворк на основі XAML, такий як WPF, WinUI або .NET MAUI, часто потрібно використовувати ObservableCollection<T> для ввімкнення зв'язування даних.

Відкрийте шаблон EntityType.t4 та знайдіть, де він генерує List<T>. Це виглядає так:

```cs
    if (navigation.IsCollection)
    {
#>
    public virtual ICollection<<#= targetType #>> <#= navigation.Name #> { get; } = new List<<#= targetType #>>();
<#
    }
```
Замініть List на ObservableCollection.

```cs
public virtual ICollection<<#= targetType #>> <#= navigation.Name #> { get; } = new ObservableCollection<<#= targetType #>>();
```

Нам також потрібно додати директиву using до коду зі скафолдом. Використання вказано у списку у верхній частині шаблону. Додайте System.Collections.ObjectModel до списку.

```cs
var usings = new List<string>
{
    "System",
    "System.Collections.Generic",
    "System.Collections.ObjectModel"
};
```
Перевірте зміни за допомогою команд зворотного проектування. Шаблони у вашому проєкті використовуються командами автоматично.

```
dotnet ef dbcontext scaffold "Data Source=(localdb)\MSSQLLocalDB;Initial Catalog=MyDB" Microsoft.EntityFrameworkCore.SqlServer
```
Якщо ви вже виконували цю команду раніше, додайте параметр --force, щоб перезаписати існуючі файли.

Якщо ви все зробили правильно, властивості навігації колекції тепер повинні використовувати ObservableCollection<T>.

```cs
public virtual ICollection<Album> Albums { get; } = new ObservableCollection<Album>();
```

## Оновлення шаблонів

Коли ви додаєте шаблони за замовчуванням до свого проєкту, створюється їхня копія на основі цієї версії EF Core. Оскільки помилки виправляються та функції додаються в наступних версіях EF Core, ваші шаблони можуть застарівати. Вам слід переглянути зміни, внесені до шаблонів EF Core, та об’єднати їх із налаштованими шаблонами.

Один зі способів переглянути зміни, внесені до шаблонів EF Core, – це використовувати git для порівняння їх між версіями. Наведена нижче команда клонує репозиторій EF Core та створить різницю між цими файлами версій 7.0.0 та 8.0.0.

```
git clone --no-checkout https://github.com/dotnet/efcore.git
cd efcore
git diff v7.0.0 v8.0.0 -- src/EFCore.Design/Scaffolding/Internal/CSharpDbContextGenerator.tt src/EFCore.Design/Scaffolding/Internal/CSharpEntityTypeGenerator.tt
```

Інший спосіб переглянути зміни – завантажити дві версії Microsoft.EntityFrameworkCore.Templates з NuGet, витягти їхній вміст (можна змінити розширення файлів на .zip) та порівняти ці файли.

Перш ніж додавати шаблони за замовчуванням до нового проєкту, не забудьте оновити пакет шаблонів EF Core до останньої версії.

```
dotnet new update
```

## Розширене використання

### Ігнорування вхідної моделі

Параметри Model та EntityType представляють один із можливих способів зіставлення з базою даних. Ви можете ігнорувати або змінювати частини моделі. Наприклад, назви навігації, які ми надаємо, можуть бути не ідеальними, і ви можете замінити їх власними під час формування коду. Інші речі, такі як назви обмежень та фільтри індексів, використовуються лише міграціями та можуть бути безпечно пропущені з моделі, якщо ви не збираєтеся використовувати міграції зі скам'янованим кодом. Так само, ви можете пропустити послідовності або обмеження за замовчуванням, якщо вони не використовуються вашою програмою.
Під час внесення таких розширених змін переконайтеся, що результуюча модель залишається сумісною з базою даних. Перевірка SQL-коду, згенерованого за допомогою dbContext.Database.GenerateCreateScript(), – гарний спосіб це перевірити.

### Класи конфігурації сутностей

Для великих моделей метод OnModelCreating класу DbContext може стати некеровано великим. Один зі способів вирішення цієї проблеми – використання класів IEntityTypeConfiguration<T>. Див. розділ Створення та налаштування моделі для отримання додаткової інформації про ці класи.
Для створення каркасу цих класів можна використовувати третій шаблон під назвою EntityTypeConfiguration.t4. Як і шаблон EntityType.t4, він використовується для кожного типу сутності в моделі та використовує параметр шаблону EntityType.

### Генерація таблиці об'єднання у зв'язках "many-to-many"

За замовчуванням процес створення каркасу не генерує сутність для таблиць об'єднання у простих зв'язках "many-to-many". Однак, бувають випадки, коли може бути необхідним явне створення таблиці об'єднань як сутності (наприклад, коли потрібен точніший контроль над згенерованим SQL-запитом).
Поведінка каркасу для кожної сутності контролюється файлом шаблону EntityType.t4. У цьому файлі є умова, яка скорочує генерацію сутностей для простих таблиць з'єднання "many-to-many". Щоб змінити цю поведінку та згенерувати об'єкт об'єднання, можна закоментувати цю умову у файлі 'EntityType.t4'.

```
<#
    // Comment this condition
    if (EntityType.IsSimpleManyToManyJoinEntityType())
    {
        // Don't scaffold these
        return "";
    }
    .  .  .    
#>
```

###  Створення каркасу для інших типів файлів

Основною метою зворотного проектування в EF Core є створення каркасу для DbContext та типів сутностей. Однак, у цих інструментах немає нічого, що вимагало б від вас фактичного створення каркасу коду.
